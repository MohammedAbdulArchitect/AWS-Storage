AWS Block & File Storage
EBS, EFS & Cloud Storage Architecture

A practitioner-focused guide to Amazon Elastic Block Store volume families, Multi-Attach, encryption workflows, and Elastic File System — including performance modes, storage tiers, and production architecture patterns.

Platform: AWS EC2 Storage EBS · EFS · KMS Encryption Production-Ready Patterns Interview Prep Included Level: Intermediate
📍 Section 01

AWS Storage Service Overview

AWS offers a layered storage stack, from raw block volumes attached to virtual machines all the way to fully managed distributed file systems. Choosing the right service hinges on access pattern, latency requirements, durability target, and cost envelope.

The Three Storage Layers

Before diving into individual services, it helps to anchor the landscape. All AWS compute-attached storage falls into one of three fundamental paradigms:

Instance Store
Ephemeral NVMe / SSD
Ephemeral

Physically attached to the host. Provides the lowest latency of any storage tier, but data is lost if the instance is stopped or terminated. Best for buffers, caches, and scratch data.

EBS
Elastic Block Store
Block · Persistent

Network-attached block device that persists independently of EC2 lifecycle. Functions like a virtual SAN volume. Supports snapshots and AZ-scoped replication.

EFS
Elastic File System
NFS · Shared

Fully managed NFSv4 file system mountable simultaneously from thousands of EC2 instances across multiple Availability Zones. Scales automatically, pay-per-GB used.

ℹ️
Key Design Principle

Instance Store → ephemeral scratch. EBS → durable single-instance block. EFS → shared, multi-instance, multi-AZ file system. Always match the storage category to the workload's durability and sharing requirements before optimising for cost.

📍 Section 03

GP2 vs GP3 — General Purpose SSD Deep Dive

The gp3 volume type supersedes gp2 in virtually every dimension. Understanding the mechanical difference between them is critical for both cost optimisation and workload right-sizing.

🔷 gp3 — Newer Generation

  • Baseline 3,000 IOPS and 125 MB/s included at no extra cost
  • IOPS can be provisioned up to 16,000 independently of volume size
  • Throughput can be provisioned up to 1,000 MB/s independently
  • Decoupling IOPS from capacity avoids over-provisioning
  • Lower cost per GB than gp2 in most regions

🔶 gp2 — Legacy Generation

  • IOPS are tightly linked to volume size: 3 IOPS per GB
  • Small volumes can burst to 3,000 IOPS via credit bucket
  • To reach 16,000 IOPS you must provision ≥ 5,334 GB
  • No independent throughput scaling
  • Max throughput capped at 250 MB/s

The gp2 IOPS Ceiling

In gp2 the performance ceiling is dictated by capacity. A workload requiring 16,000 IOPS must be backed by at least 5,334 GB of storage, regardless of whether the capacity itself is needed. gp3 eliminates this constraint entirely, making it the default choice for new deployments.

✅
Best Practice

Migrate existing gp2 volumes to gp3 for any workload that does not require burst credit behaviour. AWS supports in-place modification with no downtime, and gp3 typically reduces storage costs by 20% or more.

📍 Section 04

Provisioned IOPS — io1 & io2 Block Express

When a workload demands sustained, predictable I/O performance beyond what gp3 can deliver — or requires sub-millisecond latency — the Provisioned IOPS family is the answer.

When to Choose Provisioned IOPS

  • Relational databases requiring sustained > 16,000 IOPS (PostgreSQL, MySQL, Oracle)
  • In-memory database transaction logs with strict write latency SLAs
  • Storage-sensitive applications on EC2 Nitro instances requiring maximum throughput
  • Any workload where IOPS consistency matters more than peak burst
io1
Provisioned IOPS SSD
4 GB – 16 TB

Max 64,000 IOPS on Nitro instances (32,000 on others). IOPS provisioned independently from size. Supports EBS Multi-Attach.

io2 Block Express
Next-Gen Provisioned IOPS
Up to 64 TB

Max 256,000 IOPS with sub-millisecond latency. IOPS-to-GB ratio of 1,000:1. Supports Multi-Attach. Purpose-built for mission-critical enterprise databases.

⚠️
Nitro Requirement for Maximum IOPS

To achieve the full 64,000 IOPS ceiling on io1, the EC2 instance must be a Nitro-based instance type. Non-Nitro instances are capped at 32,000 IOPS regardless of the provisioned value on the volume. Always verify instance family when planning high-IOPS deployments.

📍 Section 05

HDD Volume Families — st1 & sc1

HDD-backed volumes trade random I/O capability for raw sequential throughput at significantly lower cost per gigabyte. They are the correct choice when the workload is read-heavy and sequential in nature.

🟡 st1 — Throughput Optimised HDD

  • Designed for large, sequential read workloads
  • Max throughput: 500 MB/s — max IOPS: 500
  • Ideal for data warehouse queries, ETL pipelines, log aggregation
  • Scales up to 16 TB per volume
  • Cannot be used as a root / boot volume

⚪ sc1 — Cold HDD

  • Lowest cost per GB of any EBS volume type
  • Max throughput: 250 MB/s — max IOPS: 250
  • Suited for data accessed only a few times per month
  • Archival datasets, compliance cold storage, DR targets
  • Cannot be used as a root / boot volume
🎯
Interview / Certification Tip

HDD volumes are characterised by throughput, not IOPS. If a scenario describes "large sequential reads of log or analytics data at the lowest possible cost," the answer is st1 or sc1. If it says "low-latency transactional," it is always SSD.

📍 Section 06

EBS Multi-Attach — Shared Block Access

Multi-Attach enables a single EBS volume to be simultaneously attached to multiple EC2 instances, enabling shared-disk clustering patterns that previously required specialised SAN hardware.

How Multi-Attach Works

A single io1 or io2 volume with Multi-Attach enabled can be mounted to up to 16 EC2 instances concurrently, all within the same Availability Zone. Every attached instance has full read and write permissions to the volume simultaneously.

Figure 2 — EBS Multi-Attach: single io2 volume attached to multiple EC2 instances in the same AZ

Critical Constraints

  • Only available for io1 and io2 volume types
  • All instances must reside in the same Availability Zone
  • Maximum of 16 instances per volume
  • Requires a cluster-aware file system (not standard EXT4 or XFS)
  • Application layer must coordinate concurrent write access to prevent data corruption
⚠️
File System Warning

Mounting a Multi-Attach EBS volume with a non-cluster-aware file system such as EXT4 or XFS across multiple hosts simultaneously will cause file system corruption. Use cluster file systems like GFS2 or Oracle OCFS2 when deploying Multi-Attach.

📍 Section 07

EBS Encryption — Data at Rest & In-Flight

EBS encryption is a zero-complexity, near-zero-latency security layer that should be enabled on every volume in production. It is handled entirely by the AWS infrastructure without any changes to the application or operating system.

What Gets Encrypted

Data at Rest

All data blocks stored on the physical volume are encrypted using AES-256. The encryption is transparent — the guest OS sees a standard block device.

Data in Transit

All I/O traffic flowing between the EC2 instance and the EBS volume is encrypted over the network path, eliminating eavesdropping risk.

Snapshots & Clones

Snapshots created from an encrypted volume are automatically encrypted. Any volume created from an encrypted snapshot inherits encryption. The chain is unbroken.

KMS Integration

Encryption keys are managed through AWS KMS (AES-256). You can use the AWS-managed default key or bring your own Customer Managed Key (CMK) for full audit trail and rotation control.

Encrypting an Existing Unencrypted Volume

EBS does not allow in-place encryption of an existing unencrypted volume. The following four-step process migrates data to an encrypted volume without application downtime if performed carefully:

  1. Create an EBS snapshot of the existing unencrypted volume. The snapshot will initially be unencrypted.
  2. Use the Copy Snapshot action and enable encryption. Select or create a KMS key. The copied snapshot will now be encrypted.
  3. Create a new EBS volume from the encrypted snapshot. The resulting volume will be encrypted by default and cannot be changed to unencrypted.
  4. Detach the original volume from the EC2 instance, attach the new encrypted volume, update any mount points or configuration references, then verify operation.

Enable the EBS Encryption by Default account-level setting. This ensures every newly created volume in the account is automatically encrypted without needing to specify a key on each creation. 📍 Section 08

Amazon EFS — Elastic File System

Amazon EFS is a fully managed, serverless NFS file system designed for high-throughput workloads that require concurrent access from many compute resources. Unlike EBS, a single EFS file system can be mounted across thousands of instances simultaneously.

Core Characteristics

  • Implements the NFSv4.1 protocol — compatible with any Linux NFS client
  • Capacity scales automatically — no pre-provisioning required; you pay only for bytes stored
  • Multi-AZ Regional file systems are highly durable (11 nines), backed by replication across three AZs
  • Compatible with Linux-based AMIs only — Windows is not supported
  • Encryption at rest with KMS, encryption in transit with TLS
  • POSIX-compliant — applications written to standard POSIX file API need zero modification
  • Priced roughly 3× the cost of a gp2 EBS volume per GB, but with zero waste since there is no pre-provisioning

Figure 4 — Amazon EFS mounted across EC2 instances in multiple Availability Zones via a single shared namespace

ℹ️
Practical Use Cases

EFS is the right choice for: CMS platforms (WordPress, Drupal) sharing a single upload directory across a web tier fleet; machine learning pipelines sharing a common dataset directory across training nodes; microservices sharing configuration or reference data; and any application whose data directory must survive instance replacement.

📍 Section 09

EFS Performance & Throughput Modes

EFS exposes two independent control surfaces: Performance Mode (latency vs. parallelism trade-off) and Throughput Mode (how throughput scales). These are set at creation time but can be changed later for Throughput Mode.

Performance Modes

⚡ General Purpose (Default)

  • Lowest per-operation latency
  • Best for latency-sensitive workloads: web servers, CMS, IDE backends
  • Recommended for the vast majority of use cases
  • Required when using Elastic throughput mode

📊 Max I/O

  • Higher aggregate throughput and IOPS for highly parallel workloads
  • Tolerates higher per-operation latency in exchange for scale
  • Targeted at big data analytics, media encoding, genome processing
  • Only available with Bursting or Provisioned throughput modes

Throughput Modes

ModeHow Throughput ScalesBest For
Bursting Grows proportionally with stored data. Burst credits accumulate during idle periods and are consumed during peak activity. Smaller file systems with spiky, infrequent workloads
Provisioned You specify a fixed throughput (e.g., 500 MB/s) regardless of file system size. Billed for the provisioned value. File systems requiring consistent throughput independently of storage used
Elastic (Recommended) AWS automatically scales throughput up and down with workload demand. Up to 3 GB/s reads, 1 GB/s writes. Unpredictable or varying workloads — simplest operational model
✅
Best Practice — Default to Elastic Mode

AWS recommends Elastic throughput mode as the starting configuration for new EFS file systems. It eliminates capacity planning for throughput, scales automatically, and you pay only for what is consumed. Switch to Provisioned only if you have measured, predictable throughput requirements that make fixed provisioning more cost-effective.

EFS Storage Tiers & Lifecycle Management

EFS Intelligent-Tiering automatically migrates files between storage tiers based on access frequency, delivering up to 90% cost savings compared to keeping all data in the Standard tier.

Lifecycle Policy Mechanics

Lifecycle policies evaluate every file in the file system independently. When a file has not been accessed for the configured threshold (e.g., 30 days), EFS moves it to the configured destination tier transparently. The file path and permissions remain unchanged — applications do not need modification.

Availability & Durability Options

✅ Regional (Multi-AZ)

  • Data replicated across ≥ 3 Availability Zones
  • Survives full AZ outage without data loss
  • Recommended for all production workloads
  • Higher monthly storage rate vs. One Zone

🔷 One Zone

  • Data stored in a single AZ only
  • Up to 47% lower cost than Regional
  • Suitable for development, testing, or easily reproducible data
  • Compatible with Standard-IA tier for additional savings
🎯
Interview / Certification Tip

EFS One Zone-IA is the cost-minimisation answer when the question specifies "development environment" or "non-critical data with infrequent access." Regional Multi-AZ is always the answer for production. Combined cost savings of up to 90% are achievable using the right tier + One Zone combination.

Cross-AZ Migration — EBS vs EFS

EBS volumes are AZ-scoped. To move data from us-east-1a to us-east-1b you must create a snapshot of the volume, then restore it in the target AZ. EFS, by contrast, presents a single namespace across AZs — no migration is necessary; an instance in any AZ simply mounts the same file system.

⚠️
EBS Snapshot I/O Impact

EBS snapshots consume I/O from the underlying volume during the backup process. Running a snapshot during peak traffic can degrade application performance. Schedule snapshots during maintenance windows or off-peak hours, and consider Amazon Data Lifecycle Manager to automate and time this correctly.

📍 Section 13

Interview Q&A — AWS Storage Scenarios

The following scenario-based questions cover the most frequently tested concepts on AWS certification exams and in cloud architect / DevOps interviews.

A financial services platform requires a database storage volume with a guaranteed minimum of 80,000 IOPS and sub-millisecond response times. The database tier runs on the latest EC2 Nitro instances. Which EBS volume type should be selected and why?

The correct choice is io2 Block Express. It is the only EBS volume type capable of delivering up to 256,000 IOPS and sub-millisecond latency. io1 is capped at 64,000 IOPS, and gp3 peaks at 16,000 IOPS — both fall short of the 80,000 IOPS requirement. The Nitro instance hypervisor is required to unlock the full io2 performance ceiling.

An engineering team wants to migrate existing gp2 EBS volumes to gp3. What is the key operational benefit, and is any downtime required?

The primary benefit is that gp3 decouples IOPS and throughput from volume size, allowing both to be scaled independently without over-provisioning storage. In gp2, IOPS grow at 3 per GB, forcing teams to increase capacity just to gain IOPS. Volume modification from gp2 to gp3 is performed in-place using the EBS Modify Volume feature — no downtime or detachment is required. The transition happens online while the volume remains attached.

A clustered Linux application requires two EC2 instances in the same Availability Zone to simultaneously read and write to a shared block device. How should this be architected?

Use an io1 or io2 EBS volume with Multi-Attach enabled. Both instances can be attached to the same volume concurrently and have full read/write access. However, a cluster-aware file system (such as GFS2 or OCFS2) must be used — standard file systems like EXT4 do not handle concurrent multi-host writes safely. The 16-instance attachment limit and same-AZ requirement must also be satisfied.

A team discovers that an existing EBS volume was created without encryption, but company policy now mandates encryption at rest for all storage. How can the volume be encrypted without losing any data?

The accepted process is: (1) create a snapshot of the existing unencrypted volume; (2) use the "Copy Snapshot" action with encryption enabled, selecting a KMS key; (3) create a new EBS volume from the encrypted snapshot; (4) detach the original volume and attach the new encrypted volume to the EC2 instance. There is no in-place encryption option for existing volumes. Enabling "Encryption by Default" at the account level prevents this situation for all future volumes.

A content management platform serves 50 web servers across three Availability Zones. All servers must read and write to a shared media directory. Which storage service is most appropriate, and what configuration minimises cost while maintaining production-grade durability?

Amazon EFS with Regional availability is the correct choice — it is the only AWS managed file system that natively supports concurrent NFS mounts across multiple AZs. To minimise cost while maintaining durability, configure a lifecycle policy to transition files not accessed within 30 days to EFS Standard-IA, and files not accessed within 90 days to EFS Archive. This alone can reduce storage costs by up to 90% for media assets that are uploaded frequently but accessed infrequently after the first few days.

Explain the difference between EBS and EFS in terms of availability zone scope and cross-AZ access patterns.

An EBS volume is tightly scoped to a single Availability Zone. It can only be attached to EC2 instances in the same AZ. Moving data to another AZ requires creating a snapshot and restoring it in the target AZ — a point-in-time copy, not live replication. EFS, by contrast, presents a single namespace accessible from any AZ within the region. Multiple mount targets allow instances across eu-west-1a, 1b, and 1c to access the same directory hierarchy simultaneously — no snapshots or data copies required.

A workload processes 500 GB daily log files in large sequential batches overnight. Storage costs are a primary constraint. Which EBS volume type is most cost-effective and why?

The st1 (Throughput Optimised HDD) volume type is the correct choice. The workload is sequential, not random — it does not require low-latency IOPS. st1 delivers up to 500 MB/s sequential throughput and 500 IOPS at a fraction of the cost of SSD volume types. It is specifically designed for throughput-intensive patterns such as log aggregation, ETL, and batch analytics. sc1 (Cold HDD) would be even cheaper but is limited to 250 MB/s throughput, which may constrain overnight batch windows.
