SECTION 01

Elastic Block Store (EBS) — Core Concepts

Elastic Block Store (EBS) is AWS's network-attached block storage service for EC2 instances. Unlike a physical drive soldered to a server, an EBS volume connects over the network fabric, making it detachable and reassignable without downtime. Data persists independently of the instance lifecycle — terminating a virtual machine does not automatically destroy the attached volume (unless explicitly configured to do so).

Think of EBS as a high-performance network USB drive: you can unplug it from one machine and re-attach it to another within the same availability zone, and your data is exactly where you left it.

Network Drive — Not a Physical Disk
EBS volumes communicate with EC2 instances over AWS's internal network. This introduces a slight latency overhead compared to physically attached disks, but it enables critical features: live detach/reattach, snapshot capabilities, and cross-instance migration without data loss.

Delete-on-Termination Behavior

When an EC2 instance is terminated, what happens to its attached EBS volumes is governed by the Delete on Termination attribute, set at instance launch time. Understanding this setting is critical for both data recovery planning and cost management.

⚙️ Default Termination Behavior
Volume TypeDefault BehaviorRecommendation
Root Volume (/dev/xvda) ✅ Deleted on termination Disable if root data must survive
Additional Data Volumes 🔒 Retained after termination Enable deletion if ephemeral usage
⚠️
Exam / Interview Tip — Delete on Termination
If asked "how do you preserve the root volume data when an EC2 instance is terminated?" — the answer is to disable the Delete on Termination attribute on the root volume before launching the instance, or modify it via the AWS CLI after launch. This is a frequently tested scenario.

SECTION 02

EBS Volume Types & Selection Guide

AWS offers several EBS volume types, each optimized for distinct workload profiles. Selecting the wrong type is a common and expensive mistake — over-provisioning IOPS on a cold storage workload, or under-provisioning throughput on a high-frequency database, both carry a cost penalty.

General Purpose SSD — GP2 vs GP3

GP3 is the current generation and should be the default choice for most workloads. It decouples IOPS and throughput from capacity, allowing fine-grained cost optimization. GP2 ties performance to volume size, which often forces over-provisioning.

Best Practice — Default to GP3
GP3 provides the same maximum IOPS as GP2 (16,000) but at a 20% lower cost per GB. It also decouples IOPS provisioning from storage size — you don't need to over-provision capacity just to get the performance you need. AWS recommends migrating all GP2 volumes to GP3 for existing workloads.

SECTION 03

EBS Snapshots — Backup, Recovery & Migration

An EBS Snapshot is a point-in-time backup of an EBS volume stored in Amazon S3. Snapshots are incremental — only the blocks that changed since the last snapshot are written, which keeps backup storage costs low. More importantly, snapshots unlock cross-AZ and cross-region volume migration capabilities.

Snapshot Architecture & Cross-AZ Migration

Snapshot Features — Archive, Recycle Bin & Fast Restore

🗄️ Snapshot Archive

Move snapshots to a cold "archive tier" for up to 75% cost reduction. Restoration from archive takes 24–72 hours — plan ahead for DR scenarios that use archived snapshots.

75% cheaper
♻️ Recycle Bin

Protects against accidental deletion. Deleted snapshots land in a Recycle Bin and can be recovered within the configured retention window (1 day to 1 year). Create a retention rule to activate this safety net.

1 day – 1 year
⚡ Fast Snapshot Restore

Forces full initialization at snapshot creation, eliminating first-use latency when launching volumes from large snapshots. Comes at significant cost premium — use only when zero-latency cold boot is required.

High Cost
💡
Architect's Tip — Snapshot Lifecycle Policy
For production workloads, always pair EBS Snapshots with an Amazon Data Lifecycle Manager (DLM) policy. This automates snapshot creation schedules, cross-region copy for DR, and archival transitions — removing the operational burden of manual snapshot management entirely.

SECTION 04

Cross-Region Disaster Recovery Strategy

EBS Snapshots stored in S3 can be copied to any AWS region. This is the foundation of a cost-effective cloud DR strategy — rather than running duplicate infrastructure 24/7, you maintain snapshots in a secondary region and provision recovery instances only during an actual failure event.

Warning — Archive Tier RTO Impact
If your DR plan uses the Snapshot Archive tier, ensure your RTO (Recovery Time Objective) allows for the 24–72 hour restore window. Archived snapshots cannot be used directly — they must be fully restored to standard tier first. For sub-4-hour RTO requirements, keep snapshots in standard tier even in the DR region.

SECTION 05

Amazon Machine Images (AMI) — Concepts & Architecture

An Amazon Machine Image (AMI) is a preconfigured template that defines everything needed to launch an EC2 instance: the operating system, software packages, security configurations, and initial storage layout. AMIs are the fundamental unit of EC2 instance standardization — rather than configuring each new instance from scratch, you encode your gold standard configuration once and stamp out identical instances on demand.

AMI Sources — Three Flavors

🟠 AWS-Managed AMIs

Maintained and patched by AWS. Amazon Linux 2023 and Amazon Linux 2 are the most widely used. Optimized for EC2 with built-in SSM agent and CloudWatch hooks. Best starting point for most workloads.

Provided by AWS
🔵 Custom AMIs

Built from your own EC2 instances after configuration. You own maintenance and patching. The primary tool for reducing boot time by baking software dependencies directly into the image layer.

Self-Managed
🟢 Marketplace AMIs

Vendor-created and optionally sold through AWS Marketplace. Common for appliances (firewalls, WAFs, database engines). May carry per-instance licensing fees on top of EC2 costs.

Third-Party

Why Custom AMIs Accelerate Deployments
A typical EC2 user-data bootstrap script might take 3–5 minutes to install a web server, monitoring agent, and custom application dependencies. If you bake all of that into a custom AMI, that installation time drops to near zero — the new instance boots with everything pre-installed. For auto-scaling groups that launch hundreds of instances under load, this difference compounds dramatically.

SECTION 06

AMI Best Practices & Governance

Custom AMIs are powerful, but without governance they become a maintenance liability. Stale AMIs carry unpatched operating systems and outdated dependencies — the opposite of what they were designed to prevent.

🏗️ AMI Build Pipeline
  • Use EC2 Image Builder to automate AMI creation on a schedule
  • Apply OS patches during each build cycle — never ship a stale AMI
  • Run CIS or STIG hardening scripts as a build pipeline stage
  • Version AMIs with semantic tags: v2.4.1-2025-06
  • Test AMI in staging before promoting to production
♻️ AMI Lifecycle Management
  • Deregister AMIs older than your retention policy (e.g., 90 days)
  • Delete associated EBS snapshots when deregistering to avoid orphaned cost
  • Copy golden AMIs to DR regions as part of your BCP
  • Share AMIs across accounts via Resource Access Manager (RAM) for org-wide standards
  • Tag AMIs with owning team, purpose, and base OS version

SECTION 07

EC2 Instance Store — High-Performance Local Disk

For workloads where EBS network latency is unacceptable, AWS offers a fundamentally different storage model: the EC2 Instance Store. This is physical NVMe storage directly attached to the host server — not a network drive, but actual hardware. The tradeoff is complete: maximum performance, zero durability.

Performance Comparison — Instance Store vs EBS

Figure 7 — IOPS comparison: NVMe Instance Store delivers orders-of-magnitude higher throughput than EBS; durability is the tradeoff

Use Cases, Tradeoffs & Architecture Decision

✅ Ideal Use Cases
  • In-memory database cache layer (Redis, Memcached overflow)
  • Temporary buffer storage for stream processing pipelines
  • Scratch space for ML model inference or batch data transformation
  • Intermediate results storage in multi-stage ETL workflows
  • Elasticsearch / OpenSearch hot node local shard storage
❌ What Instance Store Cannot Do
  • Store data that must survive instance termination or stop
  • Serve as a primary database data volume
  • Replace EBS for OS root volumes (no snapshot capability)
  • Provide data redundancy — hardware failure means data loss
  • Participate in EBS multi-attach or cross-AZ sharing
🚨
Warning — Ephemeral Storage
Instance Store is ephemeral. Data is permanently lost on instance stop, termination, or underlying hardware failure. If your application writes critical data to instance store paths (commonly /mnt/instance-store), you must implement your own replication or flushing mechanism to durable storage (EBS or S3) before any shutdown event.
💡
Exam / Interview Tip — Instance Store Recognition
Any exam question describing a need for "extremely high IOPS", "hardware-attached local disk", or "sub-millisecond storage latency" on EC2 is pointing toward EC2 Instance Store. Conversely, any requirement for "durable", "persistent", or "survives restart" means EBS.

Cloud Storage Retention Tiers — Hot, Cool & Cold

Modern cloud storage architecture separates data by access frequency and cost sensitivity into three thermal tiers. EBS, S3 Standard, S3-IA, and S3 Glacier each occupy a specific tier. Designing a correct tiering strategy can reduce storage costs by 60–80% without compromising data availability for production workloads.

SECTION 10

Scenario-Based Interview Questions & Answers

The following questions are drawn from real cloud architect and DevOps engineering interview scenarios involving EC2 storage design decisions.

Q1: Your company runs a financial transaction processing system on EC2. The database requires 200,000 IOPS and sub-millisecond response times. The data must survive instance restarts. What storage architecture would you recommend?
Recommended Architecture: Use io2 Block Express EBS volumes with provisioned IOPS. io2 Block Express supports up to 256,000 IOPS with sub-millisecond latency and maintains full durability — data survives stops and reboots. Instance Store is ruled out because the data durability requirement makes it unsuitable (data is lost on stop/terminate).

For the OS: gp3 root volume. For the database: io2 Block Express data volume with provisioned IOPS and Multi-Attach if required.
Q2: A data science team runs nightly ML model training jobs on EC2. The jobs generate 500GB of intermediate tensor data that is only needed during the training run. How would you minimize storage cost while maximizing I/O performance?
Optimal Solution: Use EC2 Instance Store for intermediate tensor data. Since the data is only needed within the training run and can be regenerated if lost, the ephemeral nature of Instance Store is not a risk. You gain NVMe-speed I/O (millions of IOPS) at no additional storage cost — Instance Store is included in the instance price.

Choose an i4i or p4d instance class which ships with NVMe Instance Store. Store model inputs/outputs on S3, and final trained models on EBS.
Q3: Your team accidentally deleted a critical EBS snapshot. Leadership needs the data recovered urgently. What AWS features could have prevented this, and what recovery options exist now?
Prevention (what should have been in place): AWS Recycle Bin with a retention rule. When enabled, deleted EBS snapshots are held in the Recycle Bin rather than permanently destroyed. Retention can be configured from 1 day to 1 year.

Recovery options now:
1. Check the Recycle Bin in the EC2 console — if a retention policy was configured, the snapshot may be recoverable.
2. If a cross-region copy was configured via DLM policy, check the DR region for a surviving copy.
3. If the EBS volume itself still exists (not deleted), create a new snapshot from it.
4. If none of the above — there is no AWS-native recovery. This is why Recycle Bin and cross-region replication are mandatory for production environments.
Q4: Your organization is deploying a fleet of 200 auto-scaling web servers. Bootstrap scripts currently take 4 minutes per instance. How do you reduce cold start time to under 30 seconds?
Solution: Custom Golden AMI Strategy
Build a custom AMI using EC2 Image Builder that includes all software dependencies (web server, monitoring agents, security tools) pre-installed and configured. Update this AMI monthly with patched packages.

In the Auto Scaling Group launch template, point to this golden AMI. New instances boot with everything pre-installed — the only startup work needed is environment-specific configuration (fetching configs from Parameter Store, setting hostname, registering with load balancer). This typically reduces cold start from 4+ minutes to under 30 seconds.

Pair with Fast Snapshot Restore (FSR) on the AMI's backing EBS snapshot to eliminate first-boot I/O latency for high-frequency scaling events.
Q5: Design a cost-optimized backup architecture for a production MySQL database with the following requirements: 15-minute RPO, 2-hour RTO for the primary region, and 24-hour RTO for a secondary DR region.
Architecture Design:

Primary Region (2-hour RTO):
• EBS gp3 volume for MySQL data with automatic EBS snapshots every 15 minutes via DLM policy → meets 15-min RPO
• Keep last 48 snapshots in standard tier (2 days coverage) → immediate restore capability meets 2-hour RTO

DR Region (24-hour RTO):
• DLM policy copies daily snapshots to secondary region
• Archive these DR copies after 30 days to EBS Snapshot Archive tier (75% cost saving)
• For a 24-hour RTO, archived snapshots are acceptable — the 24–72 hour restore window fits the requirement

Cost levers: Standard tier for recent 48 hours, archive for everything older, DLM automates entire lifecycle without manual intervention.
