Object Storage, Security & Static Hosting

A practitioner-focused guide to Amazon Simple Storage Service — covering bucket design, access control patterns, static web hosting, and object versioning with real-world architecture examples.

📍 Section 01

Amazon S3 Fundamentals

Amazon Simple Storage Service (S3) is a foundational AWS object storage platform designed for virtually unlimited scalability. It serves as the backbone of countless cloud-native applications, data pipelines, and backup workflows across the globe.

Why S3 Matters

S3 is not simply a file-storage service. It is an elastic, globally resilient object repository that integrates deeply with almost every AWS service. Whether you're distributing static content, ingesting analytics data, or building a disaster recovery pipeline, S3 is typically somewhere in the architecture.

Common Use Cases

Backup & Archive
Durability 11-9s

Store on-premises or cloud backup copies with automated lifecycle transitions to Glacier for long-term retention at minimal cost.

Disaster Recovery
Cross-Region

Replicate critical data across AWS regions so that a regional outage does not impact data availability or business continuity.

Data Lake & Analytics
Big Data

Store structured and unstructured datasets at exabyte scale and query them with Athena, Redshift Spectrum, or EMR without provisioning servers.

Static Asset Hosting
Web Scale

Serve HTML, CSS, JavaScript, images, and video to end users globally — with optional CloudFront CDN for low-latency delivery worldwide.

Hybrid Cloud Storage
Storage GW

Extend on-premises storage seamlessly to S3 using AWS Storage Gateway, enabling tiered archival without modifying existing applications.

Software Distribution
Global CDN

Host application binaries, firmware packages, and update manifests at unlimited scale — accessed by clients via pre-signed or public URLs.

ℹ️
S3 as a Platform Primitive

More than a storage service, S3 is a foundational primitive for AWS integrations. Services such as CloudTrail, Config, Athena, Lambda, and SageMaker all use S3 as their default data store. Mastering S3 design patterns is therefore essential for any AWS architect or developer.

📍 Section 02

Buckets, Objects & Key Structure

Understanding how S3 organises data — through buckets and object keys — is the first step toward designing efficient, well-structured storage hierarchies.

Buckets — The Container Layer

A bucket is a logical namespace that groups related objects. Each bucket is assigned to a specific AWS region at creation time, even though the S3 console presents a unified global view across all regions and accounts.

🌍 Global Namespace (Classic)

  • Bucket names must be globally unique across all AWS accounts and all regions
  • Once reserved, a name cannot be used by anyone else
  • Recommended for publicly accessible buckets or CDN origins
  • Harder to find available short names in competitive namespaces

🏢 Account-Regional Namespace (New)

  • AWS automatically appends an account-ID and region suffix to enforce uniqueness
  • You can freely choose any readable short name for your bucket
  • Reuse the same logical name across multiple regions within your account
  • Recommended approach going forward for internal and cross-account automation

Bucket Naming Rules

  • 3–63 lowercase characters — letters, numbers, and hyphens only
  • Must begin with a lowercase letter or digit
  • Cannot be formatted as an IP address (e.g., 192.168.10.1 is invalid)
  • Must not begin with the prefix xn--
  • Must not end with the suffix -s3alias
  • No uppercase, no underscores, no consecutive hyphens

Object Keys — The Addressing Model

Every piece of data stored in S3 is an object, uniquely identified by its key. The key is the full logical path of the object within the bucket.

🗂️ Object Key Structure
# Key of a top-level object
reports/annual-summary-2024.pdf

# Key breakdown
Prefix   → reports/
ObjectName → annual-summary-2024.pdf

# Nested "folder" keys (folders are a visual abstraction only)
logs/eu-west-1/app-server-01/2024-06-15.log

# S3 has no true directories — the entire path is the key
✅
Best Practice — Key Prefix Design

Structure your key prefixes to align with your access patterns. For analytics, partition by date (year=2024/month=06/day=15/) to enable efficient partition pruning with Athena. For application data, use entity type and region as top-level prefixes to simplify lifecycle policy targeting.

⚠️
Multi-Part Upload Requirement

Any object exceeding 5 GB must use the multi-part upload API — single-PUT uploads for files above 5 GB will fail. AWS recommends enabling multi-part upload for all objects above 100 MB to improve throughput and allow resumable uploads on network interruptions.

📍 Section 03

Amazon S3 Security Model

S3 provides a multi-layered security framework. Understanding when to use each control is critical for building least-privilege architectures that are both secure and maintainable.

Security Control Layers

IAM Policies
Identity-Based Control
User/Role Attached

Attached to IAM principals (users, groups, roles). Define which S3 API actions a principal may perform across any bucket.

Bucket Policies
Resource-Based Control
Bucket Attached

JSON policies attached directly to the bucket. Support cross-account access, public access grants, and conditional logic. Most common control mechanism.

Block Public Access
Account / Bucket Guard
Override Layer

An independent safety layer that overrides bucket policies and ACLs to prevent unintended public exposure. Can be enforced at the account level via SCP.

Object ACLs
Fine-Grain (Legacy)
Object Attached

Per-object access lists, predating bucket policies. Disabled by default on new buckets. Avoid unless integrating with legacy S3-aware systems that expect ACL semantics.

Encryption at Rest
SSE-S3 / SSE-KMS / DSSE
Data Layer

Server-side encryption using AWS-managed keys (SSE-S3), customer-managed KMS keys (SSE-KMS), or dual-layer KMS encryption (DSSE-KMS) for compliance requirements.

VPC Endpoint Policies
Network Layer Control
Network Attached

Restrict S3 access to requests originating from a specific VPC. Prevents data exfiltration paths that bypass corporate network controls.

Access Decision Logic

When a principal makes an S3 API request, AWS evaluates all applicable policies using the following logic:

flowchart TD REQ([API Request Received]) --> BLOCK{Block Public Access
Settings Enabled?} BLOCK -->|Yes & request is public| DENY1([❌ Access Denied]) BLOCK -->|No| IAM{IAM Policy
Allows Action?} IAM -->|Explicit Deny| DENY2([❌ Access Denied]) IAM -->|Allow or No Statement| RES{Bucket / Resource
Policy Allows?} RES -->|Explicit Deny| DENY3([❌ Access Denied]) RES -->|Allow| ALLOW([✅ Access Granted]) RES -->|No statement & IAM allows| ALLOW IAM -->|No statement| NOSTA{Resource Policy
Grants Allow?} NOSTA -->|Yes| ALLOW NOSTA -->|No| DENY4([❌ Access Denied])

Figure 1 — S3 access evaluation order: Block Public Access → IAM Policy → Resource/Bucket Policy

🎯
Interview / Certification Tip

The golden rule: an explicit Deny anywhere in the policy evaluation chain always wins — no Allow can override it. A principal needs both the IAM permission and the absence of a resource-level Deny to successfully access an S3 object. Remember: Block Public Access is evaluated first, before any policy.

📍 Section 04

Bucket Policies — Structure & Patterns

Bucket policies are JSON documents that define resource-level access control for an entire bucket or specific object prefixes. They are the primary mechanism for granting cross-account access and configuring public read access for web hosting.

Policy Document Structure

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowPublicRead",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-web-content-bucket/*"
    }
  ]
}
ElementPurposeExample Values
VersionPolicy language version — always use 2012-10-17"2012-10-17"
SidOptional statement identifier for human readability"AllowPublicRead"
EffectWhether to Allow or Deny the listed actions"Allow" / "Deny"
PrincipalThe IAM entity this statement applies to"*", "arn:aws:iam::123456789012:root"
ActionThe S3 API operations to permit or deny"s3:GetObject", ["s3:Put*","s3:Delete*"]
ResourceThe bucket or object ARN the statement targets"arn:aws:s3:::bucket-name/*"
ConditionOptional conditions that must be true for the statement to applyIP range, VPC, MFA present

Common Bucket Policy Patterns

Pattern 1 — Public Read (Static Website)

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::portal-assets-prod/*"
  }]
}

Pattern 2 — Cross-Account Read Access

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::987654321098:role/DataConsumerRole"
    },
    "Action": ["s3:GetObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::shared-reports-bucket",
      "arn:aws:s3:::shared-reports-bucket/*"
    ]
  }]
}

Pattern 3 — Enforce Encryption on Upload

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Deny",
    "Principal": "*",
    "Action": "s3:PutObject",
    "Resource": "arn:aws:s3:::secure-data-store/*",
    "Condition": {
      "StringNotEquals": {
        "s3:x-amz-server-side-encryption": "aws:kms"
      }
    }
  }]
}
⚠️
Block Public Access Must Be Disabled First

Before a public-read bucket policy takes effect, you must disable the Block Public Access setting at the bucket level. This is a deliberate two-step mechanism: AWS requires an explicit acknowledgement that you intend to expose the bucket publicly. Never disable Block Public Access on buckets holding sensitive company data.

IAM & Access Patterns

Different principals require different access mechanisms. Choosing the wrong approach leads to either over-privilege or operational friction. The following patterns cover the four most common S3 access scenarios.

Figure 2 — Four S3 access patterns: IAM user policy, EC2 instance role, Lambda execution role, and cross-account bucket policy

✅
Best Practice — Never Embed Access Keys

For compute resources (EC2, ECS, Lambda, Glue), always use IAM Roles rather than embedding static access keys. Roles provide time-limited, automatically-rotating credentials through the EC2 metadata service and are revocable at any time without application changes.

📍 Section 06

Static Website Hosting on S3

Amazon S3 can serve static websites — HTML, CSS, JavaScript, and media — directly from a bucket endpoint without provisioning web servers. Combined with CloudFront, this pattern supports high-traffic, globally distributed web properties.

How Static Hosting Works

When you enable static website hosting on a bucket, AWS assigns it a regional endpoint in one of two formats depending on the region:

🌐 Website Endpoint Formats
# Format A (older regions, dot separator)
http://<bucket-name>.s3-website.<region>.amazonaws.com

# Format B (newer regions, hyphen separator)
http://<bucket-name>.s3-website-<region>.amazonaws.com

# Example
http://portal-assets-prod.s3-website.eu-central-1.amazonaws.com

Setup Steps

  1. Create the bucket and upload your static files (index.html, CSS, JS, images). Ensure the bucket name matches the intended domain if using Route 53.
  2. Navigate to Bucket → Permissions → Block Public Access and disable the block to allow public reads. Acknowledge the confirmation prompt.
  3. Create and attach a Bucket Policy granting s3:GetObject to Principal: "*" for all objects (/* suffix on the resource ARN).
  4. Go to Bucket → Properties → Static Website Hosting, enable it, set the index document (e.g., index.html) and optional error document (e.g., 404.html).
  5. Copy the generated Bucket Website Endpoint URL from the Properties pane and verify it loads your site in a browser.
  6. (Optional) Create a CloudFront distribution pointing to the S3 website endpoint and configure a custom domain via Route 53 for production use.
⚠️
403 Forbidden After Enabling Hosting?

A 403 error after enabling static website hosting almost always means one of two things: (1) the Bucket Policy granting public read has not been applied yet, or (2) Block Public Access is still enabled. Check both settings — the bucket policy alone is insufficient if Block Public Access is still on. Both must be configured correctly in sequence.

Static vs Dynamic — When to Use S3 Hosting

✅ Good Fit for S3 Static Hosting

  • Single-page applications (React, Vue, Angular built output)
  • Marketing landing pages and documentation sites
  • Pre-rendered content (Hugo, Jekyll, Next.js static export)
  • Developer portals and API reference docs
  • Error pages and maintenance splash pages

❌ Not Suitable for S3 Static Hosting

  • Server-side rendering (PHP, Node.js, Python backends)
  • Applications requiring authentication at the web server layer
  • Content requiring server-to-server API calls before response
  • WebSocket connections
  • Applications writing data back to the file system

Object Versioning

S3 Versioning maintains a complete history of every change to every object in a bucket. It is a critical safeguard against both accidental deletions and unintended overwrites.

How Versioning Works

Once versioning is enabled on a bucket, S3 assigns a unique Version ID to every object uploaded. Uploading an object with the same key as an existing object does not overwrite it — instead, S3 stores both versions, with the new upload becoming the current version.

config.yaml — v2

config.yaml — DELETE MARKER
config.yaml — v2
config.yaml — v1)] G([Restore: remove delete marker]) --> H[(config.yaml — v2 visible again)]

Figure 3 — Object versioning lifecycle: upload, overwrite, soft-delete via delete marker, and restore

Key Versioning Behaviours

  • Objects that existed before versioning was enabled receive a Version ID of null
  • Suspending versioning does not delete existing versions — it only stops creating new ones
  • A regular DELETE on a versioned object creates a delete marker — the underlying versions are preserved
  • To permanently delete a specific version, you must DELETE the object with the Version ID specified
  • MFA Delete can be enabled to require multi-factor authentication before permanently deleting object versions

✅ Benefits of Versioning

  • Protection against accidental overwrites and deletes
  • Enables point-in-time recovery for individual objects
  • Satisfies compliance requirements for immutable audit trails
  • Integrates with S3 Replication for cross-region or cross-account copies
  • Pairs with lifecycle policies to auto-expire old versions and control cost

⚠️ Cost Considerations

  • Each version counts toward storage billing — costs grow as versions accumulate
  • Large, frequently updated objects can accumulate significant version storage
  • Lifecycle rules should be configured to expire non-current versions after a retention period
  • Incomplete multi-part uploads also consume storage — enable AbortIncompleteMultipartUpload rules

Lifecycle Rule Example — Expire Old Versions

{
  "Rules": [{
    "ID": "expire-old-versions",
    "Status": "Enabled",
    "Filter": { "Prefix": "logs/" },
    "NoncurrentVersionExpiration": {
      "NoncurrentDays": 90
    },
    "AbortIncompleteMultipartUpload": {
      "DaysAfterInitiation": 7
    }
  }]
}
✅
Best Practice — Version All Production Buckets

Enable versioning on every production bucket as a baseline control. Pair it with a lifecycle policy that expires non-current versions after your retention period (e.g., 90 days) to prevent unbounded storage cost growth. For critical data, also enable MFA Delete to protect against both accidental and malicious permanent deletions.

📍 Section 09

Interview Q&A — Scenario-Based

These questions reflect common patterns in cloud architect, DevOps engineer, and solutions architect interviews, as well as AWS certification exams.

A development team accidentally deleted a critical configuration file from an S3 bucket. The file is not recoverable from any other source. How should the architecture have been designed to prevent this?

The bucket should have had versioning enabled. With versioning active, deleting an object only creates a delete marker — the underlying data versions are retained. The deleted file could be recovered by either removing the delete marker or restoring the desired version. As an additional control, MFA Delete should be enabled on critical buckets to require a second factor before permanently removing any object version.

An EC2 application needs read access to a private S3 bucket. A junior engineer suggests creating an IAM user and embedding the access keys in the application's environment file. Why is this problematic, and what is the correct approach?

Embedding static IAM user credentials in an environment file or application code is a significant security risk. If the file is committed to a code repository, exposed in logs, or accessed by an attacker, the credentials provide permanent access until manually rotated. The correct approach is to create an IAM Role and attach it to the EC2 instance as an instance profile. The application then retrieves temporary credentials automatically through the EC2 Instance Metadata Service (IMDS) — no keys need to be stored anywhere, and credentials are automatically rotated every few hours.

A company hosts a React-based single-page application in S3 and distributes it via CloudFront. After deploying a new build, some users continue to see the old version. What is the root cause and how is it resolved?

The root cause is CloudFront edge cache serving the previously cached version of index.html and JS/CSS bundles. CloudFront does not automatically detect S3 object changes — it serves cached copies until the TTL expires. The resolution is to create a CloudFront invalidation for the affected paths (e.g., /* or /index.html) after each deployment. A more robust strategy is to use content-hashed filenames for JS/CSS assets so new files have new keys, while only index.html (with a short TTL) requires invalidation.

Your organisation has regulatory requirements to retain audit logs for 7 years. The logs are accessed only within the first 30 days after creation and never accessed thereafter. Design a cost-optimal S3 storage strategy.

Use an S3 lifecycle policy with the following tiers: store logs in S3 Standard for the first 30 days (active access period). Transition to S3 Standard-IA at day 30 (occasional retrieval, lower cost). Transition to S3 Glacier Deep Archive at day 90 (long-term retention, very rare access, lowest possible cost). This combination satisfies the 7-year retention requirement while reducing storage costs by 90%+ compared to keeping everything in Standard. Additionally, enable S3 Object Lock in compliance mode to prevent any modification or deletion during the retention period, satisfying immutability requirements.

A bucket policy grants s3:GetObject to "Principal": "*", but users are still receiving 403 errors when accessing objects via the public URL. What should you check first?

The first thing to verify is the Block Public Access setting at the bucket level. Even with a bucket policy that explicitly grants public access, if any of the four Block Public Access settings are enabled, AWS will override the policy and deny all public requests. Disable "Block all public access" (or the specific relevant setting) at the bucket level, acknowledge the security warning, and then verify that the bucket policy is correctly formed — specifically that the Resource ARN ends with /* (objects) not just the bucket ARN (which would only apply to s3:ListBucket).

Explain the difference between a pre-signed URL and a public S3 object URL. When would you use each?

A public object URL (e.g., https://bucket.s3.region.amazonaws.com/file.pdf) is accessible by anyone without authentication, provided the bucket policy grants GetObject to "Principal": "*". Use this for truly public content such as a website's static assets. A pre-signed URL embeds time-limited, cryptographically signed credentials from the generating IAM principal into the URL itself. It allows a specific object to be accessed without making the bucket public, and the URL expires after a configurable duration (default 15 minutes, up to 7 days with SigV4). Use pre-signed URLs for private file downloads, secure form uploads, or sharing a specific resource temporarily without changing bucket permissions.
