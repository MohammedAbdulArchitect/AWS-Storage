🔁 Core Concept

Object Versioning in Amazon S3

S3 versioning provides a built-in safety net by preserving every iteration of an object stored in a bucket. When versioning is active, overwriting an object does not erase the previous copy — it stacks a new version on top. This makes accidental overwrites, deletions, and ransomware-style attacks fully recoverable.

How Versioning Works

Once you enable versioning on a bucket, every PUT operation that targets an existing object key creates a new version ID. Objects uploaded before versioning was enabled receive a version ID of null. Objects uploaded after enabling versioning carry a unique alphanumeric version ID.

✅
Best Practice

Enable versioning on buckets that store business-critical data, configuration files, or static website assets. The storage cost overhead is minimal compared to the recovery insurance it provides. Pair versioning with lifecycle rules to auto-expire non-current versions after a retention window.

Delete Markers vs Permanent Deletes

Versioning introduces a nuanced delete model. Understanding the difference is critical for designing a reliable recovery strategy.

🗑️ Soft Delete (Delete Marker)

  • Triggered by deleting an object without specifying a version ID
  • AWS places a delete marker as the latest version
  • Underlying versions remain intact in the bucket
  • Object appears deleted when listing the bucket
  • Fully reversible — remove the delete marker to restore

❌ Permanent Delete

  • Triggered by deleting an object with a specific version ID
  • That particular version is erased permanently
  • Cannot be undone — no recovery path
  • Other versions of the same key remain unaffected
  • Use with caution; typically requires elevated IAM permissions
⚠️
Critical Warning

Permanent deletion by version ID is irreversible. Restrict the s3:DeleteObjectVersion IAM action to a small set of privileged roles. Consider enabling S3 Object Lock with MFA Delete on compliance-sensitive buckets to add an additional approval gate before any version can be erased.

Restoring a Previous Version

To roll back to an earlier state, you have two paths:

  • Remove the delete marker — when an object was soft-deleted, identify the delete marker version ID in the bucket listing (with "Show versions" enabled) and permanently delete the marker itself. The previous version automatically becomes the current, active version.
  • Delete a newer version — when an object was overwritten and you need to revert, permanently delete the unwanted version ID. The next-most-recent version becomes current.
  • Copy an old version — copy a specific version to the same key via the AWS Console, CLI, or SDK. This creates a new version identical to the historical one without discarding any intermediate versions.

🌐 Replication

S3 Replication — CRR & SRR

Amazon S3 Replication automatically copies objects from a source bucket to one or more destination buckets. The mechanism runs asynchronously in the background and supports both cross-region and same-region scenarios, each serving distinct architectural goals.

🌍 Cross-Region Replication (CRR)

  • Source and destination buckets in different AWS regions
  • Reduces read latency for geographically distributed users
  • Satisfies regulatory and data-residency compliance requirements
  • Enables replication across separate AWS accounts
  • Supports disaster recovery strategy with geographic isolation

🏠 Same-Region Replication (SRR)

  • Source and destination buckets in the same AWS region
  • Aggregate log files from multiple source buckets into one
  • Mirror production data into isolated test or staging environments
  • Lower data transfer cost than CRR
  • Can also span multiple AWS accounts within same region

Prerequisites for Enabling Replication

  • Enable versioning on both the source and destination buckets — replication will not function without versioning active on each side.
  • Attach a suitable IAM role granting the S3 service s3:GetObject on the source and s3:ReplicateObject on the destination.
  • Define a replication rule (via the Management tab → Replication Rules) that specifies the rule scope (all objects or a prefix/tag filter) and the destination bucket ARN.
  • Optionally configure delete marker replication and replica modification sync within the replication rule settings.
ℹ️
Important Behaviour

When you enable a replication rule, only new objects written after that moment are replicated automatically. To replicate objects that already exist in the source bucket, you must separately trigger an S3 Batch Replication job. This also handles any objects that previously failed replication due to transient errors.

📋 Key Rules

Replication Behaviour & Constraints

Several behavioural nuances in S3 Replication affect how you design your data pipeline. Understanding these prevents unexpected gaps in your replicated data.

📌 Core Replication Rules
  • New objects only by default — enabling a replication rule replicates future PUTs only. Pre-existing objects in the source bucket are not touched.
  • S3 Batch Replication for back-fill — run a one-time Batch Replication job to copy pre-existing objects, or to re-replicate previously failed objects.
  • Version IDs are preserved — the version ID of the replicated object in the destination bucket exactly matches the source. This is critical for audit trail integrity.
  • Delete markers are optional — by default, deleting an object in the source does NOT propagate the delete marker to the replica. You must explicitly enable Delete Marker Replication in the rule settings.
  • Permanent deletes are never replicated — deleting a specific version ID in the source never replicates to the destination. This is a deliberate safeguard against accidental or malicious mass-deletion cascades.
  • No chaining — if Bucket-A replicates to Bucket-B, and Bucket-B replicates to Bucket-C, objects from Bucket-A do not reach Bucket-C through this chain. Each rule operates independently.
🎓
Exam Tip — Delete Semantics

Exam questions often probe this distinction: delete markers CAN be replicated (if the rule is configured), but version-specific permanent deletes are NEVER replicated. This asymmetry is intentional — it protects replica buckets from cascading destructive operations originating in the source account.

💾 Storage Tiers

Amazon S3 Storage Classes

S3 offers seven distinct storage classes, each engineered for a specific access pattern and cost profile. Choosing the right class — or automating the selection — can reduce storage costs by up to 90% compared to storing everything on S3 Standard.

Durability vs Availability

Before comparing classes, two foundational metrics matter:

🛡️ Durability — 11 Nines (99.999999999%)

Durability describes the probability that an object is not lost. At 11 nines, you would statistically expect to lose a single object if you stored 10 million objects for 10,000 years. This figure is identical across all storage classes.

📡 Availability — Varies by Class

Availability describes how reliably you can access data. S3 Standard targets 99.99% (≈53 min/year downtime). Lower-tier classes accept reduced availability in exchange for lower cost. Design your applications accordingly.

Storage Class Comparison

WARM S3 Standard-IA 99.9% 30 days Per GB retrieved Disaster recovery, backup data
WARM S3 One Zone-IA 99.5% 30 days Per GB retrieved Secondary backups, re-creatable data

ARCHIVE Glacier Deep Archive 99.99% 180 days Per GB retrieved Long-term retention, compliance vaults

Millisecond retrieval

Access archived data in milliseconds. Ideal for medical images or compliance records accessed quarterly. Minimum 90-day retention.

🔄 Glacier Flexible
1 min–12 hr retrieval

Expedited: 1–5 min | Standard: 3–5 hr | Bulk (free): 5–12 hr. Best for non-urgent archive restoration at minimal cost.

🧊 Deep Archive
12–48 hr retrieval

Standard: 12 hr | Bulk: 48 hr. Lowest storage cost in S3. Minimum 180-day retention. Built for regulatory and compliance vaults.

S3 Intelligent-Tiering

Intelligent-Tiering eliminates the need to manually choose a class when access patterns are unpredictable. S3 automatically moves objects between tiers based on observed access frequency, incurring a small monthly monitoring fee per object but no retrieval charges.

Best Practice — When to Use Each Class

Use S3 Standard for active production data. Switch to Standard-IA for DR snapshots accessed rarely. Use Intelligent-Tiering when you cannot predict access patterns. Move compliance archives to Glacier Deep Archive after a configurable retention period using Lifecycle rules — this is where the biggest cost savings are realised.

⚙️ Automation

Lifecycle Policy Automation

Rather than managing storage classes manually for each object, S3 Lifecycle Policies let you define time-based rules that automatically transition objects between tiers. This is the primary mechanism for implementing tiered data retention at scale with zero operational overhead after initial configuration.

Example: Multi-Tier Lifecycle Policy

The configuration below illustrates a typical lifecycle policy for enterprise backup data. Adjust the day thresholds to match your data retention requirements and SLA agreements.

# S3 Lifecycle Rule — example via AWS CLI
aws s3api put-bucket-lifecycle-configuration \
  --bucket prod-backup-origin-acme \
  --lifecycle-configuration '{
    "Rules": [{
      "ID": "TieredRetentionPolicy",
      "Status": "Enabled",
      "Filter": { "Prefix": "" },
      "Transitions": [
        { "Days": 30,  "StorageClass": "STANDARD_IA" },
        { "Days": 60,  "StorageClass": "INTELLIGENT_TIERING" },
        { "Days": 90,  "StorageClass": "GLACIER_IR" },
        { "Days": 180, "StorageClass": "GLACIER" },
        { "Days": 365, "StorageClass": "DEEP_ARCHIVE" }
      ],
      "NoncurrentVersionTransitions": [
        { "NoncurrentDays": 30, "StorageClass": "STANDARD_IA" }
      ],
      "NoncurrentVersionExpiration": { "NoncurrentDays": 365 }
    }]
  }'
📝
Tutorial Note

Lifecycle rules operate on both current and non-current (versioned) object states. Use NoncurrentVersionTransitions to also tier out old versions of versioned objects — otherwise old versions remain in S3 Standard and continue accumulating cost.

Lifecycle Policy Configuration via Console

  • Navigate to your S3 bucket → Management tab → Lifecycle rules.
  • Click Create lifecycle rule, name it (e.g. TieredRetentionPolicy), and set the scope (all objects or a specific prefix/tag).
  • Under Lifecycle rule actions, select Transition current versions of objects between storage classes.
  • Add transition steps — for example: Standard-IA at 30 days → Glacier Flexible Retrieval at 180 days.
  • Review the Timeline summary diagram rendered by the console to verify the intended transitions, then save the rule.
⚡ High-Performance

S3 Express One Zone

S3 Express One Zone is a specialised, single-AZ storage class engineered for workloads that demand the absolute lowest latency and highest throughput from object storage. It operates through a new bucket type called a directory bucket, which differs architecturally from standard S3 buckets.

⚡ Key Characteristics
  • Single-digit millisecond latency — consistent low-latency access at scale, not just median-case.
  • 100,000s requests/second throughput — handles extreme burst and sustained high-rate workloads.
  • ~10× faster than S3 Standard — achieved by co-locating the directory bucket in the same AZ as your compute (EC2, SageMaker, EMR).
  • ~50% lower storage cost — despite the performance advantage, the per-GB cost is lower than S3 Standard due to simplified replication (single AZ only).
  • Lower availability tolerance — single-AZ means if that AZ has an outage, access is interrupted. Not suitable for multi-AZ resilience requirements.
  • Directory bucket architecture — uses a flat namespace with a directory-style path model, different from the hierarchical key model of standard buckets.
🎯 Ideal Use Cases
  • AI / ML model training (e.g. Amazon SageMaker training jobs)
  • High-performance computing — simulation data, checkpoint files
  • Financial modelling — tick-data processing, risk calculations
  • Interactive analytics (Amazon Athena, EMR, AWS Glue)
  • Media processing pipelines with high frame/segment throughput
🚫 Not Suitable For
  • Data requiring multi-AZ durability guarantees
  • Long-term archive or infrequent access data
  • Cross-region replication targets
  • Static website hosting
  • Compliance workloads requiring geographic redundancy
🎓
Interview Tip — Express One Zone vs One Zone-IA

These are both single-AZ but serve opposite use cases. One Zone-IA is for infrequently accessed, recreatable data where cost savings matter. Express One Zone is for latency-critical, high-throughput compute workloads where speed and request rate are the priority. They are not interchangeable.

🎯 Scenario Q&A

Interview & Scenario-Based Questions

These scenario-based questions reflect the type of design and configuration discussions you encounter in cloud architecture interviews and AWS certification exams.

Your team accidentally overwrote a critical configuration file stored in S3. How do you recover the previous version with minimal disruption?
If versioning is enabled on the bucket, the overwrite creates a new version rather than destroying the old one. Navigate to the object in the S3 console, enable "Show Versions," identify the version ID corresponding to the last known-good state, and either: (a) delete the newer version to promote the previous one back to current, or (b) copy the historical version to the same key using the AWS CLI — aws s3api copy-object --copy-source bucket/key?versionId=XXXX --bucket bucket --key key. If versioning was not enabled, recovery is not possible from S3 itself, which is why enabling versioning proactively on all business-critical buckets is essential.
A junior engineer deleted a folder from a versioned S3 bucket. A week later it was discovered the deletion was a mistake. How would you restore it?
In a versioning-enabled bucket, "deleting a folder" creates delete markers on all objects under that prefix. To restore, list all objects under the prefix with aws s3api list-object-versions --bucket <name> --prefix <folder/>, filter for delete markers, and batch-delete those markers using aws s3api delete-object --version-id <delete-marker-id>. Once the delete markers are removed, the objects reappear at their previous version. For large folders, use S3 Batch Operations to process the deletions at scale.
Your organisation needs to replicate S3 data from a production bucket in Mumbai to a DR bucket in Singapore, but pre-existing objects are not being replicated. Why?
S3 Cross-Region Replication (CRR) only replicates objects written after the replication rule is created. Pre-existing objects are excluded by default. To replicate them, trigger an S3 Batch Replication job targeting the source bucket with the manifest filtered by the prefix or date range you want. This also handles any objects that failed replication due to transient errors. Going forward, once the rule is active, all new PUT operations replicate automatically.
A compliance team requires that all objects in a regulatory archive remain accessible for 7 years but are accessed at most once per year. Which storage class minimises cost while meeting this SLA?
S3 Glacier Deep Archive is the optimal choice. It offers the lowest per-GB storage cost across all S3 classes with a 180-day minimum retention and retrieval times of 12 hours (Standard) or 48 hours (Bulk). For data accessed at most once per year, the retrieval cost is negligible. Implement this via a Lifecycle rule that transitions objects to Deep Archive after the initial retention window in a higher tier expires. Combine with S3 Object Lock (Compliance mode) to enforce WORM protection for the 7-year period.
Why would you choose S3 Express One Zone over S3 Standard for an ML training pipeline, despite the reduced availability?
ML training workloads are typically compute-bound and benefit enormously from low-latency, high-throughput storage access during the training run. S3 Express One Zone delivers single-digit millisecond latency and supports hundreds of thousands of requests per second when the directory bucket is co-located in the same AZ as the training instances (e.g. SageMaker). The data written during training is generally ephemeral — checkpoints and intermediate outputs can be re-generated if lost — so single-AZ availability is an acceptable trade-off. The ~50% cost saving over S3 Standard also reduces the per-training-run cost significantly at scale.
You have replication configured from Bucket-A to Bucket-B, and separately from Bucket-B to Bucket-C. Will objects in Bucket-A automatically reach Bucket-C?
No. S3 replication does not chain. Objects replicated from Bucket-A to Bucket-B are treated as new objects in Bucket-B, but they will not be forwarded to Bucket-C by the Bucket-B-to-Bucket-C rule. Each replication rule operates independently and only on objects uploaded directly to its source bucket. To propagate objects from Bucket-A to Bucket-C, you must set up an explicit replication rule directly from Bucket-A to Bucket-C.
