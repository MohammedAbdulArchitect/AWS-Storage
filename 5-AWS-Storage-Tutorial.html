📍 Section 01

Storage Class Transitions & Lifecycle Rules

Object storage platforms offer several storage tiers, each with different cost and retrieval characteristics. Rather than moving objects between tiers by hand, lifecycle rules let you automate the journey of an object from the moment it's created until the moment it's deleted.

flowchart LR A[Standard] --> B[Standard-IA] A --> C[Intelligent-Tiering] B --> C C --> D[One-Zone-IA] D --> E[Flexible Retrieval / Glacier] D --> F[Deep Archive] E --> F

Figure 1 — Permitted storage-class transition paths inside a lifecycle policy

💡
Two Building Blocks of a Lifecycle Rule

Transition actions move an object to a cheaper class after N days (e.g. "move to Standard-IA after 60 days, then to Glacier after 180 days"). Expiration actions delete an object, an old version, or an incomplete multi-part upload after N days (e.g. "delete access logs after 365 days" or "abort multi-part uploads older than 14 days").

Rules can be scoped to an entire bucket, to a specific key prefix (a sub-path), or to objects carrying a specific tag — which lets one bucket host several independent lifecycle strategies side by side.

💼
Worked Scenario

An application generates a full-resolution photo plus a thumbnail. The thumbnail is cheap to regenerate and only needs to survive 60 days; the original must stay instantly retrievable for those same 60 days, after which a multi-hour retrieval delay is acceptable. Design: store both under one bucket using two prefixes — originals/ stays on the Standard class with a transition to Glacier Flexible Retrieval after 60 days; thumbnails/ goes to One-Zone-IA with an expiration action that deletes it after 60 days.

A second common scenario combines lifecycle rules with versioning: a 30-day instant-recovery window for deleted objects (handled by the delete marker that versioning creates), followed by a rule that moves non-current versions to Standard-IA and later to Glacier Deep Archive for up to 365 days of recoverability.

🏆
Best Practice — Let Analytics Pick the Numbers

Don't guess the "after N days" thresholds. Built-in storage-class analytics tools (covering the Standard and Standard-IA classes only — not One-Zone-IA or Glacier) produce a daily-updated report with access-pattern statistics. Allow 24–48 hours after enabling analytics before the first useful data appears, then use it to tune or build new lifecycle rules.

📍 Section 02

Configuring a Lifecycle Rule — Step by Step

Lifecycle rules are created from the bucket's management settings. A single rule can combine up to five independent actions:

Action 1
Transition current versions

Move the live/current version of an object between storage classes on a schedule.

Action 2
Transition non-current versions

Move superseded (overwritten) versions of versioned objects to cheaper tiers.

Action 3
Expire current versions

Delete the live version of an object after it reaches a defined age.

Action 4
Permanently delete non-current versions

Hard-delete old versions once they are no longer useful for recovery.

Action 5
Clean up incomplete uploads & markers

Remove expired delete markers and abort stalled multi-part uploads.

timeline title Sample Lifecycle Timeline for a Versioned Bucket Day 0 : Object created (current version) Day 30 : Transition to Standard-IA Day 60 : Transition to Intelligent-Tiering Day 90 : Transition to Glacier Instant Retrieval, Non-current versions move to Glacier Flexible Retrieval Day 180 : Transition to Glacier Flexible Retrieval Day 365 : Transition to Glacier Deep Archive Day 700 : Current version expires, Non-current versions permanently deleted 
# Example: lifecycle policy snippet (JSON)
{
  "Rules": [
    {
      "ID": "tiered-retention-policy",
      "Filter": { "Prefix": "originals/" },
      "Status": "Enabled",
      "Transitions": [
        { "Days": 30, "StorageClass": "STANDARD_IA" },
        { "Days": 90, "StorageClass": "GLACIER" }
      ],
      "Expiration": { "Days": 700 }
    }
  ]
}
📝
Notes

Once a rule is reviewed and confirmed, it runs entirely in the background — there is nothing to operate day-to-day. The console (or API) will always show a visual timeline preview before you commit the rule, which is the easiest way to sanity-check the transition order.

📍 Section 03

Requester Pays Buckets

By default, the bucket owner pays for both storage and the data-transfer costs generated whenever someone downloads an object. Requester Pays flips the second half of that bill onto whoever is downloading the data.

flowchart LR subgraph Default["Default Billing"] direction LR O1["Bucket Owner"] -->|pays storage + transfer| B1["Bucket"] R1["Requester"] -->|downloads, free| B1 end subgraph RP["Requester Pays Enabled"] direction LR O2["Bucket Owner"] -->|pays storage only| B2["Bucket"] R2["Authenticated Requester"] -->|pays transfer cost| B2 end

Figure 2 — Billing responsibility shifts for data transfer when Requester Pays is enabled

⚠️
Authentication Is Mandatory

The requester must be an authenticated, signed-in identity — never anonymous — because the platform needs an identity to bill for the download. This makes Requester Pays a great fit for sharing large public-interest datasets across accounts without absorbing everyone else's bandwidth cost.

📍 Section 04

Event Notifications

Object storage platforms emit events for activity such as object creation, removal, restoration, or replication. Subscribing to these events lets you react automatically — for example, generating a thumbnail the instant an image is uploaded.

flowchart LR S3["Object Storage Bucket"] --> N1["Notification Topic (Pub/Sub)"] S3 --> N2["Message Queue"] S3 --> N3["Serverless Function"] S3 --> EB["Central Event Bus"] EB --> D1["18+ Downstream Services"] EB --> D2["Workflow Orchestration"] EB --> D3["Streaming Pipelines"]

Figure 3 — Event fan-out from a bucket to direct targets and to a central event bus

Topic
Pub/Sub Notification Topic

Fan an event out to many subscribers at once. Requires a resource access policy on the topic authorizing the bucket to publish.

Queue
Message Queue

Buffer events for asynchronous, decoupled processing. Also requires a resource access policy granting the bucket send-message rights.

Function
Serverless Function

Invoke custom code directly on each event. Requires a resource policy granting the storage service invoke permission.

Event Bus
Central Event Bus Integration

Every bucket event lands here automatically, unlocking rule-based routing, advanced filtering by metadata/size/name, multi-destination delivery, replay, and archival.

💡
Key Concept — Resource Policies, Not Identity Roles

Notification delivery is authorized with a resource-based access policy attached to the topic, queue, or function — not an identity role attached to the storage service. This mirrors how a bucket policy itself works: the permission lives on the resource being written to.

Delivery is typically near-instant — within seconds — though it can occasionally take a minute or longer under load. Notifications can be filtered by event type as well as by key prefix/suffix (e.g. only .jpg uploads).

📍 Section 05

Wiring Up a Notification — Walkthrough

The following walkthrough wires object-creation events from a bucket named media-platform-events-prod into a message queue, so a downstream worker can generate thumbnails.

  1. Create the bucket. A fresh bucket, media-platform-events-prod, is created in the target region.
  2. Open the notification settings. Under the bucket's properties, two options exist: configure individual notifications to specific targets, or simply forward all events to the central event bus.
  3. Create the queue first. A message queue, media-thumbnail-queue, is provisioned before the notification rule, since the rule needs a target to point at.
  4. Define the notification rule. Name it, optionally scope it with a prefix/suffix, select All object create events (or a narrower subset), and choose the queue as the destination.
  5. Fix the access-denied error. The first save attempt typically fails because the queue doesn't yet trust the bucket. A resource policy must be attached to the queue allowing the bucket's ARN to send messages.
  6. Re-save and verify. Once the policy is in place, the rule saves successfully, and a test connectivity message immediately appears in the queue.
  7. Trigger a real event. Uploading an object (e.g. coffee-cup.jpg) produces a queue message containing an ObjectCreated:Put event type and the object's key — exactly the payload a thumbnail-generation worker would consume.
🔒
Warning — Scope the Access Policy

For demonstration purposes a permissive "allow any sender" policy gets things working quickly, but in production the queue's resource policy should be scoped to the exact bucket ARN, not left open to any source.

Outcome: the bucket, the queue, and the access policy together form a working event pipeline — any object creation now lands as a structured message ready for downstream automation.

📍 Section 06

Performance Optimization

Object storage scales automatically to a very large number of requests, typically returning the first byte in roughly 100–200 ms. Per-prefix request limits are generous: around 3,500 write-type requests/sec (PUT/COPY/POST/DELETE) and 5,500 read-type requests/sec (GET/HEAD), and there is no limit on the number of prefixes a bucket can have.

📝
What "Per Prefix" Really Means

A prefix is everything between the bucket name and the object's filename. region-a/site-01/report.csv and region-a/site-02/report.csv are two different prefixes, each independently entitled to the full request-rate allowance. Spreading reads evenly across four prefixes can yield roughly 22,000 read requests/sec in aggregate.

Multi-Part Upload

Recommended above 100 MB and required above 5 GB, multi-part upload splits a large file into chunks that upload in parallel, then reassembles them server-side once every part has arrived.

flowchart LR F["Large File"] --> P1["Part 1"] F --> P2["Part 2"] F --> P3["Part 3"] F --> P4["Part N"] P1 --> S["Object Storage"] P2 --> S P3 --> S P4 --> S S --> R["Reassembled Object"]

Figure 4 — Multi-part upload parallelizes a large file as independent parts

Transfer Acceleration

For long-haul uploads/downloads, transfer acceleration routes traffic through the nearest of 200+ edge locations over the public internet, then carries it the rest of the way across the provider's private backbone network — minimizing the public-internet hop and maximizing the private, low-latency one. It is fully compatible with multi-part upload.

flowchart LR C["Client — Region X"] -->|public internet, short hop| E["Nearest Edge Location"] E -->|private backbone network| B["Bucket — Region Y"]

Figure 5 — Transfer acceleration minimizes public-internet distance traveled

Byte-Range Fetches

Reading large objects can also be parallelized by requesting specific byte ranges concurrently — and a failed range can be retried independently, improving resilience. The same technique lets you fetch just a known header segment (e.g. the first 50 bytes) instead of downloading the entire object.

flowchart LR Obj["Large Object"] --> R1["Range: bytes 0-999"] Obj --> R2["Range: bytes 1000-1999"] Obj --> R3["Range: bytes 2000-end"] R1 --> Cl["Client (parallel reassembly)"] R2 --> Cl R3 --> Cl

Figure 6 — Parallel byte-range GET requests speed up large downloads

🏆
Best Practice

Combine techniques: use multi-part upload for large outbound transfers, transfer acceleration for cross-region/cross-continent traffic, and byte-range fetches when only a portion of an object — or maximum parallel throughput on a download — is needed.

📍 Section 07

Batch Operations

Batch operations apply a single action — encrypting unencrypted objects, copying between buckets, updating ACLs/tags, restoring from cold storage, or invoking a custom function — across a large list of existing objects in one managed job.

flowchart LR Inv["Storage Inventory Report"] --> Query["Query Engine (filter objects)"] Query --> List["Filtered Object List"] List --> Job["Batch Job: Action + Parameters"] Job --> Done["Objects Updated + Completion Report"]

Figure 7 — Building a filtered object list, then running it through a batch job

💡
Why Not Just Script It?

A managed batch job adds automatic retry handling, progress tracking, completion notifications, and reporting — capabilities you would otherwise have to build yourself in a custom script.

A classic exam-style use case: generate an inventory report, filter it with a query engine down to every unencrypted object, then feed that filtered list into a batch job that encrypts everything at once.

📍 Section 09

Scenario-Based Interview Q&A

Work through each scenario before reading the answer — these mirror the style of questions asked of cloud storage and data-protection engineers.

A media company stores millions of thumbnails that are cheap to regenerate but rarely accessed after 60 days, alongside source images that must stay instantly retrievable. How do you design the lifecycle policy?

Separate the two by prefix. Keep source images on the Standard class with a transition to a cold-archive tier after 60 days (accepting a multi-hour retrieval delay thereafter). Move thumbnails to a low-cost, single-zone tier immediately and set an expiration action to delete them after 60 days, since they can be regenerated on demand from the source image.

Why would a dataset owner enable Requester Pays instead of just sharing the bucket normally?

Sharing a large public-interest dataset normally leaves the owner paying every download's data-transfer cost, which becomes unsustainable at scale. Requester Pays shifts that transfer cost onto each authenticated downloader while the owner continues paying only for storage — making it economically viable to share large datasets broadly.

Your event notification to a message queue keeps failing to save with an access-denied style error. What's the most likely cause and fix?

The queue does not yet have a resource-based access policy authorizing the bucket to publish messages. The fix is to attach a policy to the queue (not a role to the bucket) explicitly allowing the bucket's ARN to send messages — the same pattern applies to topics and functions used as notification targets.

A 50 GB file needs to move from an on-premises data center to a bucket on the other side of the world as fast as possible. What two features would you combine?

Multi-part upload (required well above the 5 GB threshold) to parallelize the transfer into chunks, combined with transfer acceleration to route the upload through the nearest edge location and across the provider's private backbone for the long-haul portion — the two are explicitly compatible with each other.

How would you find and remediate every unencrypted object across a 50-million-object bucket without writing a custom crawler?

Generate a storage inventory report to enumerate every object, run it through a query engine to filter down to objects lacking server-side encryption, then pass that filtered list into a managed batch operation configured to apply encryption — gaining built-in retry handling, progress tracking, and a completion report for free.

Leadership wants a single view of storage cost and risk across 40 AWS-style accounts without building custom tooling. What would you recommend, and what's the catch?

An organization-wide storage analytics/lens service, using its default multi-account, multi-region dashboard for an immediate baseline view of summary, cost-optimization, and data-protection metrics at no extra cost. The catch: the free tier only retains 14 days of history and lacks activity/status-code metrics — if leadership wants 15 months of retention or per-prefix granularity, the advanced (paid) tier is required.
