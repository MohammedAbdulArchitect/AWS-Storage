AWS S3 Security Deep Dive
Encryption · CORS · MFA Delete

A complete guide to securing Amazon S3 data — server-side and client-side encryption mechanisms, default encryption with bucket policies, Cross-Origin Resource Sharing, and Multi-Factor Authentication Delete — built for cloud and data protection professionals.

📍 Section 01

Object Encryption — Overview

Amazon S3 supports four distinct mechanisms for protecting object data at rest. Choosing the right one depends on who needs to own and manage the encryption key, and how much operational overhead you are willing to accept.

SSE-S3
Server-Side Encryption, S3-Managed Keys
Enabled by default

AWS owns and rotates the key. Encryption uses AES-256. Zero configuration required.

SSE-KMS
Server-Side Encryption, AWS KMS-Managed Keys
User-managed key

You create and control the CMK. Every use is logged in CloudTrail for full auditability.

SSE-C
Server-Side Encryption, Customer-Provided Keys
Key never stored

You supply the key with every request over HTTPS. AWS discards it immediately after use.

Client-Side Encryption
Encrypt Before Upload
Fully client-managed

Data is encrypted and decrypted entirely outside of AWS using a client-side library.

💼
Exam Tip — Know the Header for Each Mode

Each server-side mode is requested with a specific HTTP header on the upload request: x-amz-server-side-encryption: AES256 for SSE-S3, x-amz-server-side-encryption: aws:kms for SSE-KMS, and a set of x-amz-server-side-encryption-customer-* headers for SSE-C. Exam questions frequently test which header maps to which mode.

📍 Section 02

SSE-S3 — S3-Managed Keys

The simplest encryption mode. AWS owns, manages, and rotates the encryption key entirely on your behalf — you never see or interact with it.

  • Encryption type: AES-256
  • Key is owned and fully managed by AWS — never exposed to the customer
  • Required header: x-amz-server-side-encryption: AES256
  • Enabled by default on every new bucket and every new object
Fig 2.1 — Client uploads to S3; S3 pairs the object with an S3-owned key and stores it AES-256 encrypted.
📍 Section 03

SSE-KMS — AWS KMS-Managed Keys

SSE-KMS hands key ownership to you via AWS Key Management Service, adding auditability and granular access control at the cost of API throughput limits.

✅ Advantages
  • You create and control the Customer Master Key (CMK) yourself
  • Every encrypt/decrypt call is logged in AWS CloudTrail for auditing
  • Required header: x-amz-server-side-encryption: aws:kms
⚠️
Warning — KMS API Quotas

Reading or writing an SSE-KMS object triggers a call to the KMS GenerateDataKey or Decrypt API. These calls count against your regional KMS quota — typically 5,500 to 30,000 requests/second depending on region (raisable via Service Quotas). High-throughput S3 workloads fully encrypted with SSE-KMS can hit this ceiling and get throttled.

Fig 3.1 — The object and the referenced KMS key are combined; every KMS call is recorded in CloudTrail.
📍 Section 04

SSE-C — Customer-Provided Keys

With SSE-C, encryption still happens server-side inside S3, but the key itself lives entirely outside of AWS and must be supplied on every request.

  • Key is managed by the customer, outside AWS
  • S3 uses the key to encrypt the object, then immediately discards it — S3 never stores your key
  • HTTPS is mandatory — the key travels as part of the HTTP headers on every request
  • To later read the object, you must supply the exact same key again
📝
Notes — Console Limitation

SSE-C cannot be configured from the AWS Management Console. It is only available through the AWS CLI or SDKs, since the console has no secure way to accept and transmit a raw customer key.

📍 Section 05

Client-Side Encryption

The client fully owns the encryption lifecycle — data is encrypted before it ever leaves the client and decrypted only after S3 returns it.

Fig 5.1 — Encryption (and later decryption) happens entirely outside S3, on the client.
📍 Section 06

Encryption In Transit (SSL/TLS)

Separate from at-rest encryption, every S3 bucket exposes both an unencrypted HTTP endpoint and an encrypted HTTPS endpoint.

  • HTTPS = encryption in transit, also called SSL/TLS
  • Most SDKs and the AWS CLI use HTTPS by default — nothing extra to configure
  • SSE-C requires HTTPS, since the customer key travels in the request headers

Force HTTPS-Only Access with a Bucket Policy

{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::data-vault-secure-01/*",
  "Condition": {
    "Bool": { "aws:SecureTransport": "false" }
  }
}

This statement denies any GetObject call where aws:SecureTransport evaluates to false — in other words, any request not made over HTTPS.

📍 Section 07

Hands-On — Configuring Encryption

A step-by-step walkthrough of enabling and switching encryption modes on a real bucket using the AWS Console.

  1. Create a bucket named data-vault-secure-01 and enable bucket versioning so that encryption changes create new object versions instead of overwriting history.
  2. Under Default encryption, select SSE-S3 initially, then create the bucket.
  3. Upload a test object, e.g. sample-image.jpg. Open its properties and confirm it shows Server-side encryption: SSE-S3.
  4. Click Edit on the object's encryption settings, switch it to SSE-KMS, and choose the AWS-managed default key aws/s3 (no extra KMS cost). Save — this creates a new object version.
  5. Under Versions, confirm two versions now exist and that the latest one shows SSE-KMS with the aws/s3 key.
  6. Open Default encryption → Edit at the bucket level to change the bucket-wide default between SSE-S3, SSE-KMS, and DSSE-KMS (double-layer KMS encryption).
📝
Notes — What's Missing from the Console

SSE-C cannot be selected anywhere in the console — CLI/SDK only. Client-side encryption is never configured in S3 at all, since AWS has no visibility into it; encryption and decryption both happen entirely on the client before/after the transfer.

📍 Section 08

Default Encryption vs. Bucket Policies

Every S3 bucket now defaults to SSE-S3 automatically, but bucket policies give you a second, stricter layer of enforcement.

  • Default encryption (SSE-S3) is applied automatically to every new bucket and object
  • You can change the bucket-wide default to SSE-KMS or DSSE-KMS
  • A bucket policy can deny any PutObject request that lacks the correct encryption header — enforcing SSE-KMS or SSE-C explicitly
  • Bucket policies are evaluated before default encryption settings
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::data-vault-secure-01/*",
  "Condition": {
    "StringNotEquals": {
      "s3:x-amz-server-side-encryption": "aws:kms"
    }
  }
}
🏆
Best Practice

Default encryption protects you against accidental unencrypted uploads, but only a bucket policy can outright reject a non-compliant upload before it lands. Use both together for defense in depth on sensitive buckets.

📍 Section 09

CORS — Cross-Origin Resource Sharing

CORS is a browser-enforced security mechanism that controls whether a web page from one origin is allowed to request resources hosted on a different origin.

What Defines an "Origin"?

An origin is the combination of scheme + host + port. For example, in https://www.cloudportal-app.com the scheme is HTTPS, the host is www.cloudportal-app.com, and the implied port is 443. Two URLs share the same origin only if all three match exactly.

Fig 9.1 — The browser sends a pre-flight OPTIONS request to Origin B; if Origin B responds with the correct Access-Control-Allow-Origin header, the real request proceeds.
💼
Exam Tip — CORS on S3

When an S3-hosted static website needs to fetch assets from a different S3 bucket (a different origin), the asset bucket must have a CORS configuration that allows the calling origin — either a specific origin URL or * for any origin.

📍 Section 10

Hands-On — Reproducing and Fixing a CORS Error

A practical walkthrough showing exactly how a cross-origin request fails, and how a CORS configuration resolves it.

  1. Create bucket portal-frontend-app-01 and host a static website with index.html, which fetches an extra page from itself first — confirm it loads successfully (same-origin).
  2. Create a second bucket in a different region, portal-assets-cdn-02, enable static website hosting, unblock public access, and attach a public-read bucket policy.
  3. Upload extra-page.html to the second bucket and confirm it loads directly via its own website endpoint.
  4. Update index.html in the first bucket to fetch() the full URL of extra-page.html on the second bucket, then re-upload it.
  5. Reload the first site and open browser developer tools — the console shows: "Cross-Origin Request Blocked: the CORS header 'Access-Control-Allow-Origin' is missing."
  6. On the second bucket, go to Permissions → CORS configuration and add a policy allowing the first bucket's website origin.
  7. Reload the first site again — the extra page now loads successfully, and the network tab shows Access-Control-Allow-Origin and Access-Control-Allow-Methods response headers.
[
  {
    "AllowedOrigins": ["http://portal-frontend-app-01.s3-website-us-east-1.amazonaws.com"],
    "AllowedMethods": ["GET"],
    "AllowedHeaders": ["*"]
  }
]

The AllowedOrigins value must exactly match the calling origin's URL — protocol, host, and no trailing slash.

📍 Section 11

MFA Delete

MFA Delete adds a second authentication factor as a guardrail against the two most destructive S3 operations.

When Is MFA Required?

OperationMFA Required?
Permanently delete an object version✅ Yes
Suspend bucket versioning✅ Yes
Enable bucket versioning❌ No
List deleted object versions❌ No
  • Versioning must already be enabled on the bucket — MFA Delete is an extension of versioning
  • Only the bucket owner (root account) can enable or disable MFA Delete
  • Cannot be toggled from the AWS Console — CLI only
Fig 11.1 — A permanent delete-version request is rejected unless a valid, current MFA device code is supplied.
📍 Section 12

Hands-On — Enabling MFA Delete

A walkthrough of enabling MFA Delete via the CLI using a root account profile, and observing its enforcement.

  1. Create bucket vault-mfa-protect-09 in us-east-2 and enable versioning.
  2. Confirm in IAM that the root account already has a virtual MFA device registered, and copy its ARN.
  3. Create temporary root access keys (only ever for this purpose), then configure a CLI profile: aws configure --profile root-mfa-admin.
  4. Enable MFA Delete using the CLI:
aws s3api put-bucket-versioning \
  --bucket vault-mfa-protect-09 \
  --versioning-configuration Status=Enabled,MFADelete=Enabled \
  --mfa "arn:aws:iam::111122223333:mfa/root-device 123456" \
  --profile root-mfa-admin
  1. Refresh the bucket's Properties → Bucket Versioning page — it now shows MFA Delete: Enabled.
  2. Upload an object, delete it (this only adds a delete marker since versioning is on), then attempt to permanently delete a specific version ID — this is rejected with an MFA-required error.
  3. To reverse, re-run the same command with MFADelete=Disabled and a fresh MFA code, then confirm the version can now be permanently deleted.
  4. Finally, deactivate and delete the temporary root access keys — they should never persist beyond this task.
⚠️
Warning — Root Account Hygiene

Using root account access keys is strongly discouraged for everyday operations. MFA Delete configuration is one of the few legitimate exceptions, since only the root account can toggle it. Always remove the temporary keys immediately after use.

📍 Section 13

Interview & Scenario Q&A

Common scenario-based questions covering this tutorial's material.

A high-throughput S3 workload using SSE-KMS is suddenly getting throttled. Why, and how do you fix it?

Every SSE-KMS read/write calls the KMS API (GenerateDataKey/Decrypt), which is subject to a per-second regional quota. At very high request rates this quota is exhausted, causing throttling. Fixes include requesting a quota increase via Service Quotas, enabling S3 Bucket Keys to reduce the number of KMS calls per object, or switching less-sensitive workloads to SSE-S3.

A static site in one S3 bucket can't load images from a second S3 bucket. What's the likely cause and fix?

This is a classic CORS issue — the browser blocks the cross-origin fetch because the asset bucket lacks the correct Access-Control-Allow-Origin header. The fix is to add a CORS configuration on the asset bucket that allows the calling site's exact origin (or *).

A teammate accidentally deleted a critical object version and it's gone permanently. How do you prevent recurrence?

Enable versioning plus MFA Delete on the bucket. With MFA Delete active, permanently deleting any version (or suspending versioning) requires a valid MFA code from the root account, which prevents accidental or unauthorized permanent loss while still allowing normal delete-marker operations.

Compliance requires that no object can ever be uploaded to a bucket without KMS encryption, even if developers forget to set headers. What do you implement?

A bucket policy that denies s3:PutObject unless the request includes x-amz-server-side-encryption: aws:kms. Bucket policies are evaluated before default encryption settings, so this rejects non-compliant uploads outright rather than silently encrypting them differently.
