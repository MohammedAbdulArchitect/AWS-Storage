S3 Access Logs

When running cloud workloads that involve sensitive data, understanding who accessed what, when, and whether they were authorized is a compliance requirement — not just a nice-to-have. S3 Access Logs provide a complete audit trail for every request made against your bucket.

What Are S3 Access Logs?

S3 Server Access Logging captures detailed records for every request made to a source bucket — including successful reads, failed writes, and denied access attempts from any AWS account. These records are delivered as log files to a designated target logging bucket within the same AWS Region.

📊
Complete Audit Trail
Logs every API call — GET, PUT, DELETE, HEAD — including the requester identity, timestamp, and HTTP status code.
🔍
Analyzable with Athena
Log files use a structured format compatible with Amazon Athena for SQL-based analysis of access patterns and anomalies.
🌍
Same-Region Target
The logging destination bucket must reside in the same AWS Region as the source bucket being monitored.

How It Works — Architecture

When enabled, all requests reaching your source bucket are captured by the S3 logging service and asynchronously written to a separate logging bucket. There is a slight delivery delay (up to a few hours for new deployments).

Figure 1 — S3 Access Log flow: client requests → source bucket → S3 logging service → dedicated log bucket → Athena for analysis
🚨
Critical Warning — Logging Loop

Never configure the source bucket and logging bucket as the same bucket. Every write to the log generates another log entry, which generates another write — an exponential loop that inflates storage costs uncontrollably. Always use a dedicated, separate bucket for logs.

Log Entry Fields

Each S3 access log record contains the following key fields. The format is space-delimited and can be queried directly with Athena using the S3 Access Logs DDL schema.

Best Practice — Athena Integration

Point Amazon Athena at your logging bucket using the official S3 access log DDL. This lets you run queries like SELECT requester, COUNT(*) FROM s3_access_logs WHERE http_status = '403' GROUP BY requester to detect unauthorized access attempts without writing custom parsers.

🔬 Section 02

Enabling S3 Access Logs — Step by Step

The following walkthrough covers the console-based approach for enabling server access logging. The bucket policy update is automatic when you configure the logging target via the console.

Prerequisites

  • Two separate S3 buckets in the same AWS Region
  • Source bucket: the bucket you want to monitor (e.g., app-assets-prod-7731)
  • Log bucket: a dedicated bucket for log delivery (e.g., s3-audit-logs-7731)
  • Sufficient IAM permissions to modify bucket properties and policies

Step-by-Step Configuration

  1. Open the S3 console and navigate to your source bucket (e.g., app-assets-prod-7731).
  2. Go to the Properties tab and scroll down to find the Server access logging panel. Click Edit.
  3. Set the toggle to Enable. A notice will appear informing you that the target bucket policy will be updated automatically.
  4. In the Target bucket field, browse or type the name of your dedicated log bucket (s3-audit-logs-7731). Optionally add a key prefix like logs/ to organize log objects.
  5. Choose a log object key format. The default format includes bucket name, date, and a random suffix. You can also select an event-time-based format for easier partitioning in Athena.
  6. Click Save changes. The log bucket's bucket policy is automatically updated to allow the S3 logging service principal to deliver objects.
  7. Perform actions on the source bucket (upload files, list objects, download). Wait up to 30–60 minutes for the first log objects to appear in your log bucket.

Auto-Generated Bucket Policy on the Log Bucket

When you enable logging via the console, AWS automatically appends a statement to the log bucket's policy similar to the following:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "S3ServerAccessLogsPolicy",
      "Effect": "Allow",
      "Principal": {
        "Service": "logging.s3.amazonaws.com"
      },
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::s3-audit-logs-7731/*",
      "Condition": {
        "ArnLike": {
          "aws:SourceArn": "arn:aws:s3:::app-assets-prod-7731"
        },
        "StringEquals": {
          "aws:SourceAccount": "112233445566"
        }
      }
    }
  ]
}
ℹ️
Delivery Latency Note

Access logs are delivered on a best-effort basis and may arrive with a delay of several minutes to a few hours. They are not delivered in real time. For near-real-time audit needs, consider CloudTrail Data Events which have lower latency but higher cost.

S3 Pre-signed URLs

Pre-signed URLs are a powerful mechanism for granting time-limited, credential-embedded access to specific S3 objects — without making the bucket or object public, and without requiring the recipient to have their own AWS credentials.

How Pre-signed URLs Work

When you generate a pre-signed URL, AWS embeds your IAM identity's authorization signature directly into the URL. Any recipient of that URL inherits your permissions for that specific object, but only for the duration you specify.

Figure 2 — Pre-signed URL lifecycle: bucket owner generates a signed URL, delivers it to recipient who uses it to access the private object without needing AWS credentials

Expiration Limits by Method

Generation MethodMaximum ExpiryTypical Use Case
AWS Management Console12 hoursQuick one-off sharing from console
AWS CLI (presign)168 hours (7 days)Batch sharing or scripted workflows
AWS SDK (Python/JS/etc.)7 days (for SigV4)Dynamic URL generation in applications

Common Use Cases

🎬
Premium Content Delivery
Gate video or file downloads behind authentication; generate a fresh pre-signed URL upon successful login — URL expires after use window.
📤
Secure Uploads
Allow an external partner to upload a file to a specific S3 key path without giving them AWS credentials or bucket-wide access.
🔄
Dynamic URL Generation
Generate short-lived URLs on-the-fly via your application backend for an ever-changing list of authorized users.
🎓
SAA-C03 Exam Tip

When a scenario asks for temporary, time-limited access to a private S3 object for a user without AWS credentials — the answer is Pre-signed URLs. Know the maximum expiration times: 12 hours for console, 7 days for SDK/CLI (SigV4). Also remember: the generated URL carries the permissions of the IAM identity that generated it, not the recipient.

🔬 Section 04

Generating Pre-signed URLs — Console & CLI

Walk through both the console-based approach and the CLI method for generating pre-signed URLs for download and upload operations.

Via the AWS Console

  1. Navigate to your private S3 bucket (one with Block Public Access enabled).
  2. Select the target object (e.g., financial-report-q3-2025.pdf). Verify it is private by noting the Object URL returns an Access Denied error in a browser.
  3. Click Object actions → Share with a pre-signed URL.
  4. Set the desired expiration window (e.g., 5 minutes, 2 hours) and click Create pre-signed URL.
  5. Copy the generated URL. Anyone with this URL can download the object until the timer expires — no AWS account required.

Via the AWS CLI

# Generate a pre-signed URL for a GET (download) — valid for 3600 seconds (1 hour)
aws s3 presign s3://app-assets-prod-7731/reports/q3-2025.pdf \
  --expires-in 3600

# Generate a pre-signed URL for a PUT (upload) — allows partner to upload a file
aws s3 presign s3://partner-uploads-8842/incoming/vendor-data.csv \
  --expires-in 86400 \  # 24 hours
  --query-string method=PUT
💡
Best Practice — Use IAM Roles for URL Generation

Pre-signed URLs generated using temporary STS credentials (from IAM Roles) expire when the underlying role session expires — even if the URL expiry is longer. For long-lived pre-signed URLs, always generate them using a dedicated IAM user with a long-term access key, or ensure the role session duration is sufficient.

🔒 Section 05

Glacier Vault Lock & S3 Object Lock

Regulatory frameworks like FINRA, SEC Rule 17a-4, HIPAA, and GDPR require organizations to store records in a tamper-proof, non-deletable state for defined retention periods. AWS provides two mechanisms: Glacier Vault Lock for archival storage and S3 Object Lock for active storage with per-object granularity.

Glacier Vault Lock — Archive-Level Immutability

S3 Glacier Vault Lock enforces a Write Once, Read Many (WORM) policy at the vault level. Once configured and locked, the policy itself cannot be modified or removed — not even by the root AWS account or AWS support.

🧊 How Vault Lock Works
  1. Archive objects into the Glacier Vault (e.g., compliance-vault-legal-2025).
  2. Create a Vault Lock Policy — a JSON policy defining retention rules, such as deny all delete operations for 7 years.
  3. Initiate the lock. You have a 24-hour validation window to test and confirm the policy before it becomes permanent.
  4. Complete the lock. From this point on, the policy is immutable — no modifications or deletions allowed by anyone.

S3 Object Lock — Granular Per-Object WORM

S3 Object Lock provides the same WORM guarantee but at the individual object version level, with two retention modes offering different flexibility levels.

⚠️
Prerequisite — Versioning Must Be Enabled

S3 Object Lock requires bucket versioning to be enabled before it can be activated. Once Object Lock is enabled on a bucket, versioning cannot be suspended. Plan this during initial bucket provisioning.

Retention Modes Compared

🔴 Compliance Mode

The strictest retention mode. No user — including the root AWS account — can overwrite, delete, or shorten the retention period of a locked object version. Retention mode itself cannot be changed once set. Use for strict regulatory compliance (SEC, FINRA, CFTC).

🟡 Governance Mode

A more flexible mode. Standard users cannot modify or delete protected objects, but privileged IAM users with the s3:BypassGovernanceRetention permission can override the retention period or delete the object. Use for internal data governance with administrative escape hatches.

Legal Hold — Independent Indefinite Protection

Beyond retention modes, S3 Object Lock supports a Legal Hold flag on individual object versions. A legal hold:

  • Protects the object version indefinitely — independent of any retention period in effect
  • Can be placed or removed by any IAM principal with the s3:PutObjectLegalHold permission
  • Is appropriate for objects involved in active litigation, regulatory investigation, or eDiscovery holds
  • Removal of the legal hold by an authorized admin resumes normal retention or deletion behavior
Figure 3 — S3 Object Lock modes: Compliance (strictest), Governance (admin override), and Legal Hold (indefinite independent protection)
🎓
SAA-C03 Exam Tip — WORM Comparison

Know the three immutability options: Glacier Vault Lock (vault-level, policy-based, permanent once locked), Object Lock Compliance Mode (no one can delete — not even root), and Object Lock Governance Mode (admins can override with s3:BypassGovernanceRetention). Legal Hold is independent of retention periods and is toggled separately with s3:PutObjectLegalHold.

S3 Access Points

As S3 bucket usage grows across multiple teams and data domains, managing a single monolithic bucket policy becomes unmanageable. S3 Access Points solve this by creating named, independently-managed endpoints each with their own access policy scoped to a specific bucket prefix.

The Problem — Bucket Policy Sprawl

Imagine a data lake bucket with three prefixes: /finance/, /operations/, and /customer-analytics/. Without access points, you'd maintain one increasingly complex bucket policy with dozens of conditional statements. As teams and data grow, this single policy becomes a security and operations liability.

The Solution — Access Point Architecture

Figure 4 — S3 Access Points architecture: three independent access points each with their own policy scoped to specific prefixes, plus an optional VPC origin for private access

Key Properties of Access Points

  • Each access point has a unique DNS hostname (e.g., finance-ap-7731.s3-accesspoint.us-east-1.amazonaws.com)
  • Access point policies are JSON-based, identical in syntax to S3 bucket policies
  • The bucket's own policy must delegate access to the access points — a simple allow statement is sufficient
  • Access points can be Internet-facing (default) or VPC-only (private)
  • VPC access points require a VPC Endpoint (gateway or interface) in the same VPC
💡
Best Practice — Security at Scale

Use access points when your S3 bucket serves more than 2-3 teams or data domains. Configure the bucket policy to simply "Allow" Effect on "s3:*" for access points in your account" — then manage all security in the individual access point policies. This makes security auditable, modular, and independently reviewable per team.

⚡ Section 07

S3 Object Lambda

Different applications consuming data from the same S3 bucket often need different versions of that data — one wants raw data, another needs PII redacted, a third requires XML converted to JSON. S3 Object Lambda lets you run serverless code to transform objects on retrieval, eliminating duplicate bucket copies for each use case.

The Challenge — Multiple Data Views, One Bucket

Without S3 Object Lambda, organizations maintain 3-5 copies of the same data in separate S3 buckets — one for raw access, one with PII stripped, one reformatted for analytics tools. This creates storage cost, synchronization complexity, and data governance challenges.

How S3 Object Lambda Works

Figure 6 — S3 Object Lambda architecture: one S3 bucket serves three applications with different data views via Object Lambda Access Points, each backed by a dedicated transform function

Common S3 Object Lambda Use Cases

🔏
PII Redaction
Strip personally identifiable information (names, phone numbers, card numbers) from records before serving to analytics or non-production environments.
🔄
Format Conversion
Convert objects from XML to JSON, CSV to Parquet, or any format — on-the-fly — without maintaining a duplicate reformatted dataset.
💧
Dynamic Watermarking
Embed per-user watermarks into images, PDFs, or documents based on the requesting identity — each user receives a uniquely marked copy.
📊
Data Enrichment
Join S3 objects with external databases (DynamoDB, RDS) on retrieval to deliver enriched records to applications without pre-processing pipelines.
ℹ️
How Object Lambda Access Points Differ from Standard Access Points

A standard Access Point simply routes requests to a bucket prefix with a scoped policy. An Object Lambda Access Point inserts a Lambda function into the retrieval path — the function fetches the object from a supporting Standard Access Point, transforms it, and returns the modified object to the caller. No raw objects are ever delivered through the Object Lambda access point unless explicitly passed through.

Scenario-Based Interview Q&A

These scenario-based questions are representative of what you'll encounter in AWS SAA-C03 exams and cloud architect interviews focused on S3 security and governance.

Your security team has flagged that the S3 bucket prod-billing-data receives hundreds of GET requests daily. They want a full audit trail of who is accessing specific objects, when, and whether any requests were denied — without using CloudTrail. What is the most cost-effective solution?

Enable S3 Server Access Logging on prod-billing-data. Configure a dedicated log delivery bucket in the same region (e.g., billing-audit-logs-9923). The logging service will deliver structured log records covering every API call including requester identity, operation, status code, and error codes. Use Amazon Athena with the S3 access log DDL schema to query the logs. This is significantly cheaper than CloudTrail Data Events at scale, though with higher delivery latency.

A partner company needs to upload a single CSV file to your S3 bucket within the next 4 hours. They do not have an AWS account. You cannot make the bucket public. What approach provides secure, time-limited access?

Generate a Pre-signed URL for a PUT operation on the specific target S3 key using the AWS CLI: aws s3 presign s3://partner-intake-7843/incoming/partner-data.csv --expires-in 14400 (14400 seconds = 4 hours). Share this URL with the partner. They can use curl or any HTTP client to PUT the file directly to S3. The URL embeds your IAM credentials and expires automatically after 4 hours — no AWS account required on their end, and the bucket remains private.

A healthcare organization must retain patient audit logs for 7 years and ensure that even their root AWS account cannot modify or delete them. Which S3 feature should they use and which retention mode is appropriate?

They should use S3 Object Lock in Compliance Mode with a 7-year retention period. Compliance Mode ensures that no user — including the AWS root account — can overwrite, delete, or shorten the retention period of locked object versions. They must first enable versioning on the bucket, then enable Object Lock at bucket creation. Each uploaded log object should be tagged with a RetentionDate lock set 7 years into the future. For added immutability at the vault level for archival copies, they could also use Glacier Vault Lock for objects that have aged beyond their active access window.

You have a single S3 bucket serving raw order data. A new legal mandate requires that your analytics pipeline only receives data with customer names and contact numbers removed. Creating a copy of every object is too expensive. What is the recommended AWS-native approach?

Use S3 Object Lambda. Create a Lambda function that receives the raw object, strips PII fields (names, phone numbers, email addresses) using regex or Comprehend PII detection, and returns the sanitized payload. Create a Standard S3 Access Point on the source bucket, then create an S3 Object Lambda Access Point linked to the Lambda function. Configure your analytics pipeline to use the Object Lambda access point URL instead of the direct bucket URL. The original data remains intact in S3 — the transformation only occurs during retrieval, with zero additional storage cost.

A large enterprise has 8 business units each needing different read/write permissions on different prefixes within the same S3 data lake. Managing the bucket policy has become a 400-line JSON nightmare. What is the cleanest architectural fix?

Replace the monolithic bucket policy with S3 Access Points. Create one access point per business unit, each with its own JSON policy scoped to the relevant prefix (e.g., Finance AP → /finance/, HR AP → /hr-records/). Simplify the bucket policy to a single delegating statement that allows any access point in the account to interact with the bucket. Each team's IAM role policy references only their access point ARN. This creates independently auditable, modular security policies — adding a new team is now a 10-line JSON addition rather than editing a shared 400-line document.

A company's legal team has placed a litigation hold on several S3 objects that may be needed as evidence. These objects must not be deleted for any reason, even after their regular 1-year retention period expires. How do you implement this in S3?

Apply an S3 Object Lock Legal Hold to each affected object version using the s3:PutObjectLegalHold API or console. A legal hold protects the object indefinitely, completely independent of any retention period or retention mode in effect. Even after the 1-year governance retention expires, the legal hold will continue to prevent deletion. When the litigation concludes, an authorized IAM principal with the s3:PutObjectLegalHold permission can remove the legal hold, at which point the object becomes subject to normal deletion procedures again.

Summary Cheat Sheet — Key Decision Points

Audit who accessed what → S3 Access Logs (cheapest) or CloudTrail Data Events (near real-time)
Temporary access without AWS credentials → Pre-signed URL
Strict immutability, no one can override → Object Lock Compliance Mode
Immutability with admin escape hatch → Object Lock Governance Mode
Litigation freeze, indefinite duration → Legal Hold
Archive immutability at vault level → Glacier Vault Lock
Multi-team bucket access, clean policy separation → S3 Access Points
On-the-fly object transformation without data duplication → S3 Object Lambda
