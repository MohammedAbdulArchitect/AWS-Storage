AWS CloudFront & Global Accelerator

A hands-on technical tutorial covering content delivery networks, edge caching, geo-restriction, cache invalidation, and global traffic acceleration on AWS.

📡 Section 01

CloudFront — Content Delivery Network (CDN)

AWS CloudFront is a globally distributed content delivery network that accelerates the delivery of web assets by caching content at edge locations close to end users. On the AWS exam, whenever you see the term CDN, think CloudFront.

Why Use a CDN?

Without a CDN, every user request — regardless of their geographic location — must travel all the way to the origin server. This introduces latency proportional to distance. CloudFront solves this by maintaining a network of hundreds of points of presence (PoPs) around the globe, including edge locations and regional edge caches. Content is served locally, reducing round-trip time dramatically.

⚡
Reduced Latency

Content is cached at edge locations near users, dramatically cutting response times globally.

🛡️
DDoS Protection

CloudFront's distributed nature absorbs volumetric attacks via AWS Shield integration.

🌍
Global Scale

Hundreds of PoPs across every major continent deliver consistent performance worldwide.

📐 CloudFront Global Edge Architecture

Figure 1 — CloudFront edge caching: US users hit a cached copy (HIT), while EU users trigger an origin fetch on first request (MISS)

💡
Tutorial Note

CloudFront also integrates with AWS WAF (Web Application Firewall) and AWS Shield for multi-layer DDoS protection at the edge — traffic never even reaches your origin during an attack.

📦 Section 02

CloudFront Origin Types

An origin is the authoritative source of your content — the backend that CloudFront fetches from when the edge cache doesn't have what a user requested. CloudFront supports multiple origin types.

🪣
Amazon S3 Bucket

Distribute and cache static files at the edge. Secured using Origin Access Control (OAC) — the bucket remains private, only CloudFront can read it.

🔒
VPC Origin (Private)

Connect directly to private ALBs, NLBs, or EC2 instances inside a VPC without exposing them publicly. NEW Recommended for production.

🌐
S3 Static Website

Use an S3 bucket configured as a static website as a custom HTTP origin. The bucket must have website hosting enabled.

⚖️
Public HTTP Backend

Any publicly accessible HTTP endpoint — a public Application Load Balancer, EC2 instance, or third-party server — can serve as a custom origin.

⚠️
Security Warning

When using a public HTTP backend (old approach), you must restrict access to CloudFront's published IP ranges at the security group level. Failing to do so leaves your origin directly exposed to the internet. The new VPC Origin method eliminates this risk entirely.

⚙️ Section 03

How CloudFront Works — Request Lifecycle

Understanding the request flow is essential for diagnosing cache behavior and configuring TTL correctly.

  1. User sends an HTTP/HTTPS request to the CloudFront domain (e.g., d3xyz123.cloudfront.net). DNS routes this to the nearest edge location.
  2. Edge location checks its local cache. If the object is present and its TTL hasn't expired → Cache HIT. Content is returned immediately.
  3. On Cache MISS, the edge location forwards the request to your configured origin server (S3, ALB, etc.) over the AWS private backbone network.
  4. Origin responds with the content. The edge location caches the response locally for future requests.
  5. Subsequent requests from the same or nearby users are served directly from the edge cache — no origin round-trip needed until TTL expires.
🔄 CloudFront Request Flow — Cache HIT vs MISS

Figure 2 — On Cache HIT the edge responds immediately; on Cache MISS the edge fetches from origin, caches the response, then returns it to the user

🔄 Section 04

CloudFront vs S3 Cross-Region Replication

These two services are often confused on AWS exams. They solve fundamentally different problems.

☁️ CloudFront (CDN)
  • Uses the Global Edge network (~450+ PoPs)
  • Content is cached at edge locations
  • Cache TTL: typically minutes to days
  • Ideal for static content serving worldwide
  • Requires no per-region setup
  • Content may be slightly stale until cache expires
  • Supports both static and dynamic content
🔁 S3 Cross-Region Replication
  • Replicates an entire S3 bucket to specific regions
  • No caching — objects are fully replicated
  • Updates are near real-time
  • Must be configured per target region
  • Ideal for dynamic content with strict consistency
  • Read-only in target regions by default
  • Used for compliance, disaster recovery, low-latency reads
🎯
Exam & Interview Tip

If the exam asks about serving static content to users globally with minimum latency → CloudFront. If it asks about replicating data for disaster recovery or compliance in specific regions → S3 Cross-Region Replication. The key differentiator is caching vs replication.

Attribute CloudFront S3 Cross-Region Replication
PurposeContent delivery / CDNData replication
ScopeAll ~450+ edge locationsSpecific target regions only
ConsistencyCached (TTL-based)Near real-time
Content TypeStatic + dynamicS3 objects only
Write SupportUpload via PUT supportedRead-only in target
Use CaseGlobal website, API accelerationDR, compliance, multi-region reads
🛠️ Section 05

Lab: Setting Up CloudFront with S3

This walkthrough demonstrates how to create a private S3 bucket and serve its content securely through a CloudFront distribution using Origin Access Control (OAC).

Prerequisites

  • An active AWS account with permissions for S3 and CloudFront
  • Sample content files ready to upload (HTML page, images)
  • Basic familiarity with the AWS Management Console

Step-by-Step Guide

  1. Create a private S3 bucket
    Navigate to S3 → Create bucket. Give it a unique name (e.g., acme-web-assets-prod-01). Leave Block all public access enabled. Do not add any public bucket policy — the bucket stays private.
  2. Upload your content
    Upload your assets — e.g., index.html, logo.png, and supporting images. These objects are private at this stage and cannot be accessed via direct S3 URLs.
  3. Create a CloudFront Distribution
    Navigate to CloudFront → Create distribution. Select the Free plan (sufficient for labs). Set the origin type to Amazon S3 and select your bucket.
  4. Enable OAC (Origin Access Control)
    When prompted, select Allow private S3 bucket access to CloudFront using the recommended OAC settings. CloudFront will automatically update your S3 bucket policy to allow only this distribution.
  5. Review the auto-generated bucket policy
    After the distribution is created, navigate to S3 → Permissions → Bucket policy. You'll see a policy granting s3:GetObject permission exclusively to your CloudFront distribution's service principal.
  6. Access content via CloudFront domain
    Copy the distribution domain name (e.g., d2abc123xyz.cloudfront.net). Open it in a browser followed by a resource path like /logo.png or /index.html.
ℹ️
Bucket Policy Example (OAC)

CloudFront automatically injects a bucket policy like the one below. Do not remove or override it.

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontServicePrincipal",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::acme-web-assets-prod-01/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/EDFDVBD6EXAMPLE"
        }
      }
    }
  ]
}
✅
Best Practice

Always use Origin Access Control (OAC) over the older Origin Access Identity (OAI). OAC supports S3 SSE-KMS encryption, all S3 storage classes, and is the AWS-recommended approach going forward.

Using ALB or EC2 as a CloudFront Origin

When your application runs on compute (EC2, containers, or behind a load balancer) rather than object storage, CloudFront can still sit in front of it — using either VPC Origins (recommended) or a public HTTP backend (legacy approach).

🔒 Section 07

VPC Origins — The Recommended Approach

VPC Origins allow CloudFront to connect directly to resources in private subnets, without requiring a public IP address or security group IP allowlisting.

🔐
Private ALB

An Application Load Balancer in a private subnet — no public DNS or public IP required.

⚡
Private NLB

Network Load Balancer for TCP/UDP workloads requiring ultra-low latency.

🖥️
Private EC2

Individual EC2 instances in private subnets can serve as direct CloudFront backends.

✅
Best Practice

Use VPC Origins as your default configuration for compute-backed origins. It eliminates the need to maintain IP allowlists, reduces attack surface, and simplifies security group management.

🗺️ Section 08

CloudFront Geo Restriction

Geo restriction allows you to control which countries can access your CloudFront distribution. Country detection is performed using a third-party geo-IP database matched against the user's IP address.

✅
Allowlist Mode

Only users from explicitly permitted countries can access content. All others receive a 403 Forbidden response.

🚫
Blocklist Mode

Users from specified countries are denied access. All other countries are permitted by default.

Common Use Cases

  • Enforcing copyright licensing restrictions (e.g., streaming media available only in licensed territories)
  • Blocking access from regions with trade compliance requirements
  • Regional content differentiation — showing different content by country
⚠️
Plan Requirement

Geographic restrictions require a Pay-as-you-Go or paid CloudFront plan. The free tier does not include this feature. Upgrade your distribution plan before configuring geo-restriction.

♻️ Section 09

Cache Invalidation

By default, CloudFront serves cached content until the TTL expires. When you update your origin and need the changes to be reflected immediately, you must explicitly invalidate the cache.

What is Cache Invalidation?

Cache invalidation instructs CloudFront to remove specific objects from all edge location caches. On the next request for those objects, the edge will fetch a fresh copy from origin and re-cache it.

♻️ Cache Invalidation Workflow

Figure 4 — Invalidation propagates through all edge locations; subsequent requests re-fetch from origin and re-cache the latest version

Common Invalidation Paths

# Invalidate everything (most common)
Path: /*

# Invalidate a specific file
Path: /index.html

# Invalidate all files in a directory
Path: /images/*
Path: /assets/css/*

# Invalidate multiple specific files
Path: /index.html
Path: /about.html
⚠️
Cost Consideration

Each invalidation request is free for up to 1,000 path invalidations per month. Beyond that, each path costs $0.005. Using /* counts as a single path. Use targeted paths in production to minimize cost.

🚀 Section 10

AWS Global Accelerator — Overview

AWS Global Accelerator is a networking service that routes user traffic through the AWS global backbone to improve performance for applications running in one or more AWS regions.

The Problem It Solves

When your application is deployed in a single region (e.g., Mumbai), users in distant regions (Tokyo, London, São Paulo) must traverse the public internet to reach it. Each router hop adds latency and introduces packet loss risk. Global Accelerator eliminates excessive public internet traversal by routing traffic to the nearest AWS edge location, then through the AWS private network directly to your application.

🌐 Global Accelerator — Traffic Flow

Figure 5 — Each user connects to the nearest edge PoP via the public internet (short hop), then traffic rides the AWS private backbone to the application

Key Features

📡
2 Static Anycast IPs

Two static global IPs are assigned to your accelerator. They never change, so clients don't need to update their IP allowlists.

❤️
Health Checks + Failover

Automatic health checks on all endpoints. Failover to a healthy region occurs in under 60 seconds — great for disaster recovery.

⚖️
Intelligent Routing

Routes to the lowest-latency edge, considering real-time network conditions — not just geographic proximity.

🛡️
DDoS Protection

Integrated with AWS Shield Standard. Two static IPs are easy to protect vs a dynamic IP range.

📡 Section 11

Unicast vs Anycast IP — The Key Concept

Global Accelerator's routing model is built on Anycast IP, which behaves differently from the familiar Unicast model.

🎯 Unicast IP (Traditional)
  • One server = one unique IP address
  • Client traffic is directed to a specific fixed server
  • If that server is geographically distant → high latency
  • Used by traditional DNS-based load balancing
🌐 Anycast IP (Global Accelerator)
  • Multiple servers share the same IP address
  • Network routing sends client to the nearest server
  • Transparent to the client — same IP, optimal routing
  • Enables single global IPs that route to closest PoP
🎯
Exam Tip

Questions about static global IP addresses that route to multiple regions → Global Accelerator using Anycast. Questions about caching content at the edge → CloudFront. The routing mechanism is the key differentiator.

⚖️ Section 12

CloudFront vs Global Accelerator

Both services use the AWS global edge network and AWS Shield for DDoS protection. Their purposes are fundamentally different — understanding the distinction is critical for the exam.

Attribute CloudFront Global Accelerator
Primary PurposeContent delivery (CDN)Application acceleration
Caching✅ Yes — edge cache❌ No — pure proxy
IP AddressesDynamic CloudFront IPs2 Static Anycast IPs
ProtocolsHTTP / HTTPS onlyTCP, UDP (any protocol)
Content Served FromEdge locations (cached)Application in AWS region
Health ChecksOrigin health checksEndpoint health checks + failover
Ideal ForWebsites, APIs, static filesGaming, IoT, VoIP, TCP apps
Client Cache IssuesPossible (IP changes)None (static IPs never change)
Regional FailoverSlower (DNS-based)Fast (<60 seconds)
💡
Decision Framework

Use CloudFront when you need edge caching, HTTP acceleration, or content delivery. Use Global Accelerator when you need static IPs, non-HTTP protocols, deterministic regional failover, or applications where caching is not applicable.

🛠️ Section 13

Lab: Deploying AWS Global Accelerator

This lab provisions a Global Accelerator with EC2 backend endpoints in two regions to demonstrate Anycast-based geographic routing.

Lab Architecture

📍 Region 1: us-east-1 (N. Virginia) — EC2 instance serving HTTP on port 80
📍 Region 2: eu-central-1 (Frankfurt) — EC2 instance serving HTTP on port 80
🌐 Global Accelerator routes each user to the nearest region automatically

Step 1 — Launch EC2 Instances in Two Regions

In each target region, launch a t3.micro Amazon Linux instance with the following user data script. Customize the region name in each instance's script:

#!/bin/bash
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd

# Get instance metadata
INSTANCE_ID=$(curl -s http://169.254.169.254/latest/meta-data/instance-id)
AZ=$(curl -s http://169.254.169.254/latest/meta-data/placement/availability-zone)

echo "<html><body>
  <h1>Response from: us-east-1</h1>
  <p>Instance: $INSTANCE_ID</p>
  <p>AZ: $AZ</p>
</body></html>" > /var/www/html/index.html
⚠️
Security Group Configuration

Each EC2 instance's security group must allow inbound TCP port 80 from 0.0.0.0/0. Global Accelerator's health checks and traffic will originate from various AWS IPs. For production, restrict this using Global Accelerator's IP ranges after setup.

Step 2 — Create the Global Accelerator

  1. Navigate to Global Accelerator → Create accelerator. Name it prod-ga-web-01. Type: Standard. Routing: IPv4.
  2. Add a listener: Protocol TCP, Port 80. Client affinity: None.
  3. Add endpoint groups for us-east-1 and eu-central-1.
  4. Under each endpoint group, add the corresponding EC2 instance as an endpoint type.
  5. Configure health checks: Protocol HTTP, Path /, Port 80, Interval 10s, Threshold 2.
  6. Wait for status to show Healthy on both endpoints (~2 minutes).

Step 3 — Validate Routing

Once provisioned, copy one of the two static Anycast IPs. Open it in a browser. You should see a response from the endpoint geographically closest to your current location. Using a VPN to connect from a different country will demonstrate the accelerator routing to the other region.

💸
Cost Warning — Clean Up After Lab

Global Accelerator charges $0.025/hour per accelerator plus data transfer fees. After completing the lab: (1) Disable the accelerator, (2) Delete the accelerator, (3) Terminate both EC2 instances in both regions.

💬 Section 14

Scenario-Based Interview Questions

Test your understanding with these real-world scenarios — the kind of questions you'll encounter in AWS certification exams and technical interviews.

Your company hosts a media streaming platform. Users in Asia report 4–5 second load times for video thumbnails hosted in S3 (us-east-1). What is the most cost-effective solution?
Deploy a CloudFront distribution with the S3 bucket as origin using Origin Access Control (OAC). CloudFront will cache the thumbnails at edge locations near Asia, reducing load times to under 100ms for subsequent requests. S3 Cross-Region Replication would copy data but doesn't cache — it's more expensive and doesn't solve the latency problem at scale.
A gaming company needs to expose two static IP addresses globally for their UDP-based game server fleet. Players must connect to the nearest region. Which service handles this?
AWS Global Accelerator. CloudFront does not support UDP traffic. Global Accelerator provides two static Anycast IPs, routes UDP traffic to the nearest PoP, then forwards it to the regional game server fleet via the AWS private backbone. The static IPs allow players to hardcode them in their firewall or client configs.
Your marketing team updated the website's hero image in S3, but users are still seeing the old image after 6 hours. Why, and how do you fix it immediately?
CloudFront is serving the old image from its edge cache because the TTL hasn't expired yet. The fix is to create a CloudFront invalidation for the affected path (e.g., /images/hero.jpg or /*). This removes the object from all edge caches. On the next request, the edge fetches the updated image from S3 and re-caches it. To prevent this in future, use versioned file names (e.g., hero-v2.jpg) — you never need to invalidate because the URL changes.
You need to prevent users from a specific country from accessing your CloudFront-distributed content due to licensing restrictions. What feature should you use?
CloudFront Geo Restriction. Configure a blocklist with the target country's country code. CloudFront uses a third-party geo-IP database to identify the user's country from their IP address and returns a 403 Forbidden response. Note that geo restriction requires a paid CloudFront plan — the free tier does not include this feature.
A solutions architect is migrating from directly exposing an Application Load Balancer to using CloudFront. How should they secure the ALB origin to prevent direct access bypassing CloudFront?
Use CloudFront VPC Origins (recommended). This keeps the ALB in a private subnet — it has no public IP and no public DNS. Only CloudFront can route to it via the VPC Origin connector. The legacy approach (setting ALB security group to allow CloudFront's public IP ranges) is fragile and harder to maintain. VPC Origins eliminate this dependency entirely and are the current AWS best practice.
Compare the failover behavior of CloudFront vs Global Accelerator when an origin region becomes unavailable.
CloudFront uses origin failover configured via origin groups — it can fall back to a secondary origin, but the process depends on DNS TTL and cache behavior. Global Accelerator performs active health checks every 10–30 seconds and automatically reroutes traffic to a healthy endpoint within under 60 seconds of failure detection. For applications requiring fast, deterministic regional failover, Global Accelerator is the superior choice.
